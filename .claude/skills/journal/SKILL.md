# /journal — 빌드 저널 갱신

## 개요
이 프로젝트의 빌드 저널(개발 일지, `docs/journal/README.md`)에 새 작업 항목을 추가한다. 목적: 면접 "가장 어려웠던 부분은?"에 바로 꺼내 쓰려고 그때그때 기록하는 것. 자동이 아니라 사용자가 `/journal` 할 때만 실행한다.

## 언제
- 사용자가 `/journal` 입력
- "저널 갱신/기록", "개발 일지 써줘", "오늘 작업 정리해서 저널에" 같은 요청

## 절차
1. **날짜 확인**: `date "+%Y-%m-%d (%a)"`를 실행해 실제 날짜·요일을 확인한다. 추측으로 적지 말 것(요일 자주 틀림).
2. **맥락 파악**: 이번 세션(또는 최근)에 *한 일 / 막혔다 푼 점·결정 / 다음 할 일*을 대화 맥락에서 뽑는다. 불확실하면 사용자에게 한 줄로 확인.
3. **작성 규칙**: 쉬운 한국어로. 어려운 말·줄임말은 뒤에 괄호로 풀이 — 예: `ADR(왜 이렇게 정했는지 이유를 적어두는 결정 기록)`.
4. **추가 위치**: `docs/journal/README.md` **맨 아래**에 `---` 구분선을 두고 새 항목을 붙인다. (오래된 게 위, 새 게 아래로 쌓임 — 한 파일 누적 방식)
5. **설계 결정 확인**: 이번 작업에 설계·기술 결정(ADR감)이 있었으면, 항목을 추가한 뒤 **"ADR도 추가할까요?"** 라고 한 번 물어본다. (이미 ADR을 만들었으면 그 사실만 알려주고 묻지 않아도 됨)
6. **면접 답변 확인**: 이번 작업이 면접 예상질문(`docs/interview-prep.md`의 ①~⑥) 답변 거리가 되면, **"면접 답변(interview-prep)으로도 한 줄 남길까요?"** 라고 한 번 물어본다.

## 항목 형식
```markdown
## YYYY-MM-DD (요일) — 한 줄 제목

### 한 일
- ...

### 막힌 점 / 결정
- ...

### 다음
- ...
```

## 흔한 실수
- 날짜를 `date` 확인 없이 적기 → 요일 틀림. 꼭 명령으로 확인.
- 어려운 용어를 풀이 없이 쓰기 → 쉬운 한국어 + 괄호 풀이.
- 새 항목을 파일 위쪽에 끼워넣기 → 반드시 **맨 아래**에 추가.
- 설계 결정이 있었는데 ADR 물어보기를 건너뛰기.
