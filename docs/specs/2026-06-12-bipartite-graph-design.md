# 설계 문서 — 이분 그래프 C-MVP (문서·개념 자동 연결)

- 날짜: 2026-06-12
- 상태: 초안 (사용자 검토 대기)
- 관련 결정 기록: [0013](../decisions/0013-bipartite-model-pivot.md)(피벗) · [0001](../decisions/0001-node-granularity.md)·[0002](../decisions/0002-node-source.md)(일부 대체) · [0009](../decisions/0009-extractor-noise-strategy.md)(추출 노이즈) · [0011](../decisions/0011-db-auth-architecture.md)(DB)
- 옛 설계: [2026-05-29-knowledge-graph-design.md](2026-05-29-knowledge-graph-design.md) — **기록용 보존**(수동 엣지 모델). 이 문서가 그 위에 피벗을 얹는다.

> 쉬운 한국어. 어려운 말은 괄호 풀이. 용어: **이분 그래프**(노드가 두 종류로 갈리고 엣지는 서로 다른 종류 사이에만 잇는 그래프), **멤버십**(소속 = "이 문서가 이 개념을 담음"), **정규화**(표기를 통일된 형태로 바꾸는 것), **degree**(노드에 붙은 연결선 수).

---

## 1. 한 줄 요약 / 핵심 가치

여러 학습 자료(`.md`·velog 글)를 넣으면 **손 하나 안 대고** 개념을 뽑아 **문서·개념 이분 그래프**를 자동으로 그려서, **흩어진 내 자료들이 *어떤 개념을 통해* 이어지는지 발견**하게 하는 도구. 핵심 가치는 그대로 **관계 발견**이되, 이번엔 *내가 안 그린 연결*이 자동으로 뜬다.

발견은 구조에서 나온다: 여러 문서가 같은 개념 노드를 가리키면 그 개념이 **다리**가 되어 문서들이 이어지고(클러스터), 한 문서에만 달린 개념·아무와도 안 엮인 문서는 **고립**으로 드러난다("아직 네 지식망과 안 엮임").

## 2. 왜 만드나 (동기·피벗 이유)

학습 기록이 흩어져 "무엇을 배웠고 어떻게 이어지는지" 한눈에 안 보인다(원동기 동일). 그런데 옛 MVP는 엣지를 **손으로** 그어 *내가 이미 아는 연결만* 그릴 수 있었다 → "발견"이라는 핵심 가치와 어긋났다. 그래서 연결을 자료 내용에서 **자동으로** 끌어내도록 피벗한다(상세 근거 [0013]). Obsidian조차 `[[링크]]`를 사람이 직접 쳐야 하지만, 우리는 안 쳐도 연결이 뜬다 — 그게 차별점이다.

## 3. 무엇을 만드나 (C-MVP 범위)

**넣는 것**

- **다중 입력 누적**: `.md` 여러 개 올리기 + velog 주소 여러 개 → 각 자료가 **문서 노드** 하나.
- **개념 자동 추출**: 기존 추출기 재사용(제목·인라인코드·볼드). 사람 채택 단계 없음.
- **표기 정규화**: 소문자·앞뒤공백·구분기호 통일 → *같은 표기 = 같은 개념 노드*로 자동 병합.
- **문서→개념 멤버십 엣지 자동 생성.**
- **노드 두 종류 시각 구분**(문서 vs 개념: 모양·색).
- 움직이는 그래프(끌기·확대축소·이동), 검색 + 거르기.
- **고립 강조**(degree 0) — 의미 재정의(아래 §4·§7).
- 저장: **기존 DB+인증 재사용**(로그인→서버 저장·복원).

**빼는 것(2차 이후 — 로드맵)**: 수동 엣지(제거 확정), 주제 수동 지정/색(채택 단계가 사라져 함께 빠짐 — §4 참조), 퀴즈·이해도·추천, LLM 일체, 개념 별칭②·임베딩, velog 추출 정교화.

## 4. 데이터 모델

```
노드(Node):  { id, kind: 'document' | 'concept', label }
엣지(Edge):  { id, source(document id), target(concept id) }   // 방향: 문서 → 개념(멤버십)
```

- **문서 노드**: `label` = 파일명 또는 velog 글 제목/주소. `id` = 출처 기반 안정 키(같은 자료 다시 넣으면 같은 노드).
- **개념 노드**: `id` = **정규화 키**(`normalize(label)`), `label` = **대표 표기**(처음 등장한 원형 — 화면엔 못생긴 정규화형이 아니라 원형을 보여줌). 같은 정규화 키면 자동으로 한 노드(여러 문서가 공유 → 그 개념을 통해 연결).
- **엣지**: 문서-개념 쌍 하나당 하나(중복 제거). 개념↔개념·문서↔문서 엣지는 **없음**(이분 그래프 규칙).
- **색**: 이제 색은 **노드 종류**(문서/개념)를 나타낸다. (← 옛 모델의 "주제=색"은 사람이 주제를 지정하던 채택 단계가 사라지며 빠짐. 클러스터는 색이 아니라 *공유 개념 구조*로 드러나므로 색을 주제에 쓸 필요가 줄었다. [0001]의 "주제=색" 원칙은 보류.)
- **고립 판정**: `isIsolated = (degree == 0)`. 문서 노드면 "아무 개념도 안 담음/공유 안 함", 개념 노드면 "한 문서에만 달려 다리 역할 못 함"은 degree 1로 약연결. (degree는 저장 안 하고 엣지에서 계산.)

## 5. 아키텍처

```
┌───────────────────────── Next.js 앱 ──────────────────────────┐
│  [브라우저(클라이언트)]                      [서버 라우트]             │
│  1. 다중 입력                                                   │
│     · .md 여러 개 올리기 ─────────────┐                          │
│     · velog 주소 여러 개 ─────────────┼──▶ /api/velog ─▶ velog   │
│                                     │     ◀── 마크다운 ──        │
│  2. 각 자료: 개념 추출(순수) ◀──────────┘                          │
│  3. 정규화 + 그래프 빌드(순수, TDD)                                 │
│       문서들 + 각자의 개념목록 → 이분 그래프(노드·엣지, 개념 병합)        │
│  4. 그래프 화면 (문서/개념 구분·검색/거르기·고립 강조)                   │
│  5. 저장 (로그인→Supabase 서버, 기존 재사용)                         │
└───────────────────────────────────────────────────────────────┘
```

velog는 CORS로 브라우저에서 직접 못 부르므로 서버 라우트가 대신 호출(기존 그대로, [0004]).

## 6. 데이터 흐름

여러 자료 입력 → (자료마다) 개념 추출 → 정규화 → **그래프 빌드**(문서 노드 + 개념 노드 병합 + 멤버십 엣지) → 그래프 화면 → 저장. 한 방향이고, 입력이 `.md`든 velog든 추출부터는 **같은 길**을 탄다. 핵심 차이: 자료가 **누적**되며 같은 개념이 여러 문서에서 나오면 그 시점에 연결이 *생겨난다.*

> **그래프 = 올린 문서 집합의 함수(파생 상태).** 진짜 입력은 "어떤 자료를 넣었나"뿐이고, 노드·엣지는 매번 그 집합에서 다시 빌드된다. 따라서 **문서를 빼면 그 문서에만 있던 개념·연결도 함께 사라진다** — 손으로 노드를 자유 편집하지 않는다(전부 자동 피벗의 정신, [0013]). 저장도 단순해진다: 문서 집합(자료 출처 + 추출 개념)을 보관하면 그래프는 언제든 다시 빌드 가능.

## 7. 모듈 분해 (각자 책임 하나)

| 모듈 | 책임 | 입력 → 출력 | 상태 |
| --- | --- | --- | --- |
| 추출기 `extractor` | 글에서 후보 개념 뽑기 | 마크다운 → 후보 목록 | **재사용** |
| 정규화 `normalize` | 표기 통일 키 만들기 | 개념 텍스트 → 정규화 키 | **신규**(순수·TDD) |
| 그래프 빌더 `graph-builder` | 문서들+개념목록 → 이분 그래프 | 문서·개념 → 노드·엣지(병합) | **신규**(순수·TDD) |
| 그래프 연산 `graph-ops` | degree·isolated | 엣지 → 판정 | **수정**(이분 재정의) |
| 그래프 상태 `graph-store` | 문서 집합 보관 + 파생 그래프(노드·엣지) | 문서 추가/제거 → 다시 빌드 | **수정** |
| 입력 `InputPanel` | 다중 .md·velog 누적 | 자료 → 문서+개념 | **수정**(한 건 교체 → 누적) |
| 그래프 화면 `GraphView` | 종류별 그리기·거르기·고립강조 | 상태 → 화면 | **수정**(kind별 표시) |
| 저장 `storage` | 서버 저장/복원 | 상태 ↔ Supabase | **수정**(스키마 변화) |
| velog 라우트 | velog 본문 가져오기 | 주소 → 마크다운 | **재사용** |

> **삭제**: 수동 엣지(점 2번 클릭=연결) 로직. **DB 스키마 변화 주의**: `nodes`에 `kind` 추가, `edges`가 문서→개념 멤버십이 됨 → **마이그레이션 필요**(구현 계획에서 다룸, dev/prod 분리 환경 [0012] 위에서).

순수 로직은 `src/lib`, 화면은 `src/components`, 페이지·라우트는 `src/app`(관심사 분리 유지). 테스트는 대상 옆(co-location), 공용 설정만 `src/test`.

## 8. 에러 처리 (바깥에서 들어오는 데이터만 검사)

검사는 바깥 데이터 세 곳 + 빈 결과에만: ① 올린 `.md` ② velog 응답(실패 시 "마크다운 직접 붙여넣기" 대체 경로, [0004]) ③ 붙여넣은 주소 ④ 추출 0개("뽑을 게 없다" 안내). 같은 자료 중복 입력 → 같은 문서 노드로 병합(새로 안 만듦). 같은 개념 → 정규화 키로 병합.

## 9. 테스트 전략

**C-MVP = 유닛(TDD) + 컴포넌트(RTL) + E2E(Playwright) + 수동 검수**, 네 층으로 분담.

- `normalize`·`graph-builder` = **유닛 TDD**(순수 함수 — 새 TDD 연습거리. "두 문서가 공유 개념으로 이어지는지"가 빌더 테스트의 핵심 골든 케이스).
- `graph-ops`(고립 재정의) = 유닛.
- `InputPanel`(다중 입력·에러·빈 상태) = **RTL 컴포넌트 테스트**(폼류 — 클릭→DOM 반응).
- **그래프 렌더(canvas) = E2E(Playwright) + 수동 검수로 분담**:
  - **Playwright(자동 게이트)** — ① 흐름 골든패스(여러 자료 업로드 → "확정된 점 N개" 증가 → canvas 등장), ② canvas 스모크(빈 화면 아님). 실제 브라우저라 jsdom과 달리 canvas를 진짜 그려, "안 그려짐/안 이어짐" 사고를 자동으로 막는다.
  - **사람(수동)** — "관계가 보기 좋게·직관적으로 드러나나"의 *판단*. 도구가 대신 못 하는 영역.
  - 시각 회귀(스크린샷 비교)는 force 레이아웃이 비결정적이라 flaky → 레이아웃 고정 비용 때문에 **선택(나중)**.
- DB 저장·인증·마이그레이션 = **수동 검수**(mock은 "통과 ≠ 동작" — HANDOFF 경고).
- 약한 게이트: `tsc` + lint.

> ⚠️ 커버리지 함정 + 도구 분담: 유닛이 다 통과해도 **화면엔 안 뜰 수 있다.** RTL은 canvas를 못 본다(jsdom 가짜 브라우저). 그래서 "그래프가 진짜 그려지나"는 **실제 브라우저를 띄우는 Playwright**가 골든패스+스모크로 메우고, "보기 좋나/맞나"만 사람이 본다. (RTL=부품 검사 / Playwright=완성앱 도로주행 / 사람=미적·의미 판단.)

## 10. 결정 현황

- ✅ **이분 그래프 피벗** ([0013]) — 노드 두 종류, 엣지 자동 멤버십, 개념 전부 자동+정규화, 수동 엣지 제거, LLM은 퀴즈 자리.
- ✅ **DB 영속 대상 = 문서 집합(원천)** ([0014](../decisions/0014-persist-document-set.md)) — 빌드된 그래프가 아니라 문서만 저장하고 복원 시 재빌드. 빌더가 바뀌어도 저장 데이터가 안 굳음(파생 모델과 일치).
- 미정/후속: 노이즈가 심하면 "2개 이상 문서 공유 개념만 노드"로 좁히기 / 개념 별칭②·임베딩 / 퀴즈 LLM 방식 — 모두 후속 ADR 후보.

## 11. 용어 풀이

- **이분 그래프**: 노드가 두 종류, 엣지는 다른 종류끼리만.
- **멤버십 엣지**: "이 문서가 이 개념을 담음"이라는 소속 연결.
- **정규화**: 비교·병합용으로 표기를 통일(화면 표시는 원형 유지).
- **순수 함수**: 같은 입력에 늘 같은 출력 → 테스트 쉬움.
- **고립/약연결**: degree 0 / degree 1 — 아직 지식망에 덜 엮인 노드.
