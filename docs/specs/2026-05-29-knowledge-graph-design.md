# 설계 문서 — 학습 지식 그래프 (MVP)

- 날짜: 2026-05-29
- 상태: 초안 (사용자 검토 대기)
- 관련 결정 기록: [0001](../decisions/0001-node-granularity.md) · [0002](../decisions/0002-node-source.md) · [0003](../decisions/0003-framework.md) · [0004](../decisions/0004-velog-fetch.md)

> 쉬운 한국어로 작성. 어려운 말은 뒤에 괄호로 풀이. 이 문서는 결정 기록(ADR)들을 하나로 합친 "무엇을·어떻게 만들 것인가" 청사진이다.

---

## 1. 한 줄 요약 / 핵심 가치

학습한 개념들을 점(노드)과 연결선(엣지)으로 그려, **따로 배운 개념들이 사실은 연결돼 있다는 걸 시각적으로 발견**하게 하는 도구. 핵심 가치는 **관계 발견**이다.

발견은 두 곳에서 나온다: (1) 손으로 그은 연결들이 모여 만드는 **전체 구조**(같은 색 덩어리, 여러 개를 잇는 다리 역할 점), (2) 연결이 비어 있거나 적은 점이 주는 **"여기를 더 이어볼까" 힌트**.

**성장 방향**: 이 MVP(1단계)는 배운 걸 되돌아보는 "지도"다. 다음 단계로 개념 점을 클릭하면 퀴즈로 이해도를 측정하는 "능동 학습 도구"로 키운다(자세히는 [roadmap.md](../roadmap.md) #1). 핵심 가치는 1단계 **관계 발견** → 그 위에 **능동 학습**을 얹는 식으로 성장한다.

## 2. 왜 만드나 (동기·문제)

- 학습 기록(주간 학습 로그, velog 블로그 글)이 흩어져 있어 **무엇을 배웠고 어떻게 이어지는지 한눈에 보기 어렵다.**
- 취업 포트폴리오 + 도구/워크플로우 학습 + 본인이 쓸 실용 도구 + 멘토 과제를 겸한다.
- 메타 목표: **면접에서 "어떻게 만들었는지"를 자세히 설명**할 수 있도록, 만들면서 결정 기록(ADR)·빌드 저널·면접 예상질문을 함께 채운다.

## 3. 무엇을 만드나 (MVP 범위)

**1차에 넣는 것**

- 학습 로그(.md 파일 올리기) + velog 글(주소 붙여넣기)에서 후보 키워드 뽑기 → 사람이 골라 개념 점으로 확정
- 점·연결선 손으로 추가/삭제
- 주제별 색깔
- 움직이는 그래프(끌기·확대축소·이동)
- 검색 + 주제별 거르기
- 연결이 하나도 없는 점(고립) 강조 — 회색·크게. (2026-06-07 변경: 처음엔 "고립·약연결(degree ≤ 1)"이었으나, 한 쌍만 이어도 양끝이 회색으로 남아 대비가 안 보였다 → "고립(degree 0)"만 강조로 좁힘. 이슈 #29)
- 브라우저 안에 저장 + JSON 내보내기/가져오기

**다음으로 미루는 것**: 무엇을·왜 미루는지의 **기준 문서는 [roadmap.md](../roadmap.md)("나중에 할 것")** 다(중복을 피하려 여기 나열하지 않음). 요약 — 이해도 측정(퀴즈 ★)·이해도 표시·다음 학습 추천·연결선 자동 추천·배포(MVP 직후 1순위)·velog 추출 정교화·DB 저장.

## 4. 데이터 모델

```
개념 점(Node): { id, label(개념 이름), topic(주제 → 색) }
연결선(Edge):  { id, source, target }      // 방향 없음 = "관련 있음"
```

- 점 종류는 한 가지(평평한 구조). 주제는 별도 점이 아니라 **색 속성**([0001]).
- 주제는 사람이 자유롭게 입력(예: "React", "DB"). **같은 주제 이름 = 같은 색**(미리 정한 색 묶음에서 자동 배정).
- "고립 강조"는 `isIsolated`(연결 수 degree == 0)로 판정하며, degree는 따로 저장하지 않고 연결선에서 계산한다.

## 5. 아키텍처

```
┌────────────────────────── Next.js 앱 ─────────────────────────────┐
│  [브라우저 화면(클라이언트)]                 [서버 라우트]                 │
│  1. 입력                                                           │
│     · 학습 로그 .md 올리기 ────────────────┐                          │
│     · velog 글 주소 붙여넣기 ──────────────┼──▶ /api/velog            │
│                                        │      └─GraphQL─▶velog    │
│                                        │      ◀── 마크다운 ──       │
│  2. 후보 뽑기 (순수 함수, TDD) ◀────────────┘                          │
│  3. 사람 확인 화면 (받기/고치기/버리기 + 주제 지정) → 점 확정                 │
│  4. 그래프 화면 (끌기·연결·색·검색/거르기·고립 강조)                         │
│  5. 저장 (브라우저 안 + JSON 내보내기/가져오기)                            │
└────────────────────────────────────────────────────────────────────┘
```

- 프레임워크 = Next.js. velog는 브라우저에서 직접 못 부르므로(CORS) **서버 라우트가 대신 호출**한다([0003]·[0004]).
- 용어: **CORS**(다른 사이트를 브라우저에서 직접 못 받게 막는 보안 규칙), **서버 라우트**(서버에서 도는 코드 한 조각), **GraphQL**(velog 서버의 데이터 창구 방식 — **우리 앱이 채택하는 기술이 아니라**, velog에 요청할 때만 그 형식으로 `fetch` 한 번 보냄. 별도 라이브러리 불필요).
- **GraphQL은 velog 쪽 사정**일 뿐, 이 앱의 스택은 그냥 **Next.js + fetch**다.

## 6. 데이터 흐름

입력(로그/velog) → 후보 뽑기 → 사람 확인 → 점 확정 → 손으로 연결 → 그래프 → 저장. 한 방향이며, 입력 출처가 무엇이든 **후보 뽑기부터는 같은 길**(섞기 흐름, [0002])을 탄다.

## 7. 모듈 분해 (각자 책임 하나)

| 모듈                      | 책임                                | 입력 → 출력                      | 의존                         |
| ------------------------- | ----------------------------------- | -------------------------------- | ---------------------------- |
| 추출기                    | 글에서 후보 키워드 뽑기             | 마크다운 → 후보 목록(+출처 위치) | 없음(순수 함수)              |
| velog 라우트 `/api/velog` | velog 글 본문 가져오기              | 주소 → 마크다운                  | velog GraphQL                |
| 파일 입력                 | .md 올려 텍스트로                   | 파일 → 텍스트                    | 브라우저 FileReader          |
| 후보 확인 UI              | 받기/고치기/버리기 + 주제 지정      | 후보 목록 → 확정된 점            | 추출기 결과                  |
| 그래프 상태               | 점·선 보관/추가/삭제                | 사용자 동작 → 상태               | 저장소                       |
| 그래프 화면               | 그리기·끌기·연결·색·거르기·고립강조 | 상태 → 화면                      | react-force-graph (ADR 0006) |
| 저장소                    | 저장/복원                           | 상태 ↔ 브라우저 저장/JSON        | 브라우저                     |

## 7.1 디렉토리 구조와 역할

§7의 "논리적 모듈"을 **실제 폴더**에 배치하는 규칙. 순수 로직은 `src/lib`, 화면(UI)은 `src/components`, 페이지·서버 라우트는 `src/app`에 둔다 — **관심사 분리**(로직과 화면을 갈라놓음). 핵심 효과: `lib`의 로직은 화면·브라우저 없이 떼어내 **단위 테스트**할 수 있다(우리가 TDD 한 부분이 전부 여기).

```
src/
├── lib/          # 순수 로직 (화면 X. UI 아님) — 단위 테스트 대상. 테스트하기 쉬움
│   ├── types.ts        # 공유 타입 (Candidate·GraphNode·GraphEdge·Graph)
│   ├── extractor.ts    # 마크다운 → 후보 키워드
│   ├── storage.ts      # localStorage 저장/복원 + JSON 내보내기/가져오기
│   ├── graph-ops.ts    # degree·isIsolated 등 그래프 연산
│   ├── graph-store.ts  # zustand 그래프 상태(점·선 보관/추가/삭제)
│   └── velog.ts        # velog 주소 파싱 + 본문 가져오기
├── components/   # React UI 조각 (화면)
│   ├── InputPanel.tsx       # .md 업로드 + velog 주소 입력
│   ├── CandidateReview.tsx  # 후보 채택 → 점 확정(주제 지정)
│   ├── GraphView.tsx        # react-force-graph 렌더·연결·색·고립 강조
│   ├── SearchFilter.tsx     # 개념 검색 + 주제 거르기
│   └── Controls.tsx         # JSON 내보내기/가져오기 + 점 삭제
├── app/          # 페이지 + 서버 라우트 (Next.js 규칙: 파일 위치 = 경로)
│   ├── layout.tsx          # 공통 레이아웃 (스캐폴딩 기본)
│   ├── page.tsx            # 메인 화면 (위 컴포넌트들을 조립)
│   └── api/velog/route.ts  # velog 본문 가져오는 서버 라우트 (CORS 우회, [0004])
└── test/         # 테스트 전용 공용 설정 (앱 빌드 X) — 모든 테스트가 공유
    └── setup.ts  # RTL의 jest-dom matcher 등록 (vitest setupFiles로 테스트 전 실행)

# 테스트 파일(.test.ts/.test.tsx)은 대상 코드 "옆에" 둠 (co-location):
#   lib/extractor.test.ts · components/CandidateReview.test.tsx …
```

| 폴더             | 역할                       | 왜 따로 두나                                    |
| ---------------- | -------------------------- | ----------------------------------------------- |
| `src/lib`        | 순수 로직·상태 (화면 아님) | 테스트하기 쉬움(순수 함수) · 재사용 · UI와 분리 |
| `src/components` | React UI 조각              | "보여주기"만 전담 → 로직과 안 섞임              |
| `src/app`        | 페이지 + 서버 라우트       | Next.js 규칙(파일 위치가 곧 URL 경로)           |
| `src/test`       | 테스트 전용 **공용 설정**  | 모든 테스트가 공유 → 어느 대상에도 안 묶임(중립). 테스트 *파일* 자체는 대상 옆(co-location) |

> 같은 Task 안에서도 자리가 갈린다 — 예: Task 6의 `parseVelogUrl`(로직)은 `lib/velog.ts`, velog 라우트는 `app/api/velog/route.ts`.
> **테스트 배치**: 한 대상을 검사하는 테스트 파일은 그 대상 옆(co-location). `setup.ts`처럼 모든 테스트가 공유하는 설정만 중립 폴더 `src/test`에.

## 8. 에러 처리 (바깥에서 들어오는 데이터만 검사, 안쪽 내 코드는 신뢰)

검사·에러 처리는 **바깥에서 들어오는 데이터 세 곳**에만 둔다: ① 올린 .md 파일 ② velog 응답 ③ 붙여넣은 주소. 그 뒤 내 코드끼리 주고받는 안쪽은 다시 검사하지 않는다.

- velog 가져오기 실패(네트워크·비공식 창구 변경) → 명확한 에러 + **"마크다운 직접 붙여넣기" 대체 경로**([0004]).
- 빈 파일·후보 0개 → "뽑을 게 없다" 안내.
- 같은 개념 중복 채택 → 합치기(이미 있으면 새로 안 만듦).

## 9. 테스트 전략

**MVP = 유닛 테스트(TDD) + 폼류 컴포넌트 RTL 테스트.**

- **lib 로직(추출기 등) = 유닛 테스트 + TDD**(순수 함수라 테스트 먼저 짜기 좋음) → 로직의 **강한 게이트**. 러너 = **Vitest**.
- **폼류 컴포넌트(후보 확인 UI 등) = RTL 컴포넌트 테스트**(Vitest + RTL + jsdom). 클릭→상태 변화를 검증. *(2026-06-04 도입 — 학습 목적 + jsdom으로 테스트 가능한 영역이라 앞당김. RTL=React Testing Library)*
- 그 밖의 MVP 게이트: `tsc`(타입체크, 약한 게이트) + lint + **사람 수동 검수**.

**나중으로 미룸** (1차엔 과해서):

- **그래프 화면(GraphView) = 수동 검수**. canvas(react-force-graph)는 jsdom이 안 그려 RTL로 시각 검증 불가 → 사람이 눈으로.
- velog 라우트 = 통합 테스트(velog 응답을 가짜로 넣어 검증).
- 전체 흐름 = E2E(Playwright) 골든패스(정상 성공 경로) 1개.

⚠️ 커버리지 함정: 추출기 유닛테스트가 다 통과해도 **그래프가 화면에 잘 그려지는지는 증명 못 한다** → MVP에선 **사람 수동 검수**로 그 부분을 메운다.

## 10. 결정 현황

- ✅ **그래프 그리는 라이브러리 = react-force-graph** ([0006](../decisions/0006-graph-library.md))
- ✅ **테스트 러너 = Vitest** ([0005](../decisions/0005-tech-stack.md))
- 현재 남은 설계 결정 없음. (채용 공고에 D3·시각화가 반복되면 그래프 화면을 D3 직접 구현으로 전환 검토 — [0006] 재검토 조건)

## 11. 용어 풀이

- **MVP**: 최소 기능 제품 — 일단 핵심만 담은 1차 버전.
- **노드/엣지**: 점/연결선.
- **순수 함수**: 같은 입력에 늘 같은 출력, 바깥 상태를 안 건드림 → 테스트하기 쉬움.
- **유닛/컴포넌트/E2E 테스트**: 부품 하나만 / 화면 조각 / 사용자처럼 전체.
- **골든패스**: 에러 없이 정상으로 끝까지 가는 성공 시나리오.
