# 0014. DB 영속 대상 = 빌드된 그래프가 아니라 "문서 집합(원천)"

- 상태: 채택
- 날짜: 2026-06-12
- 관련: [0013](0013-bipartite-model-pivot.md)(이분 피벗) · [0011](0011-db-auth-architecture.md)(DB 아키텍처) · [0012](0012-dev-prod-db-separation.md)(dev/prod) · spec [2026-06-12](../specs/2026-06-12-bipartite-graph-design.md) §6·§10

> 용어: **파생 상태**(다른 데이터에서 매번 계산해 만드는 값 — 여기선 그래프), **진실의 원천(source of truth)**(데이터의 정본이 되는 단 하나의 출처).

## 맥락 (왜 이 결정이 필요한가)

[0013] 피벗으로 **그래프 = 올린 문서 집합의 파생**이 됐다(spec §6: 노드·엣지는 매번 `buildGraph(documents)`로 만든다). 로그인 사용자의 데이터를 Supabase에 영속해야 하는데(멀티기기·백업, [0011]), **"무엇을 저장하느냐"**를 정해야 한다 — 파생 *결과*(빌드된 노드·엣지)냐, *원천*(문서 집합)이냐.

## 고려한 선택지

- **A. 문서 집합(원천)을 저장** — `documents`(자료 출처 + 추출 개념). 복원 시 `buildGraph`로 그래프를 다시 만든다.
  - 장점: 빌더가 바뀌어도 저장 데이터가 안 굳음, 진실의 원천이 하나. 단점: 새 테이블·RPC·매핑이 필요.
- **B. 빌드된 그래프(노드·엣지)를 저장** — 옛 `nodes`/`edges` 테이블·`replace_graph` RPC·`rowsToGraph` 재사용.
  - 장점: 기존 인프라 그대로(마이그레이션 작음). 단점: 빌더 변경마다 저장 그래프가 옛 로직 결과로 굳어 데이터 마이그레이션 필요, 원천이 둘(현 문서 상태 + 저장 그래프)로 갈림.

## 결정

**A. 문서 집합(원천)을 저장한다.** 빌드된 그래프는 영속하지 않고, 복원 시 문서에서 다시 빌드한다.

## 근거 / 무엇을 얻고 무엇을 포기했나

- **빌더 재변경에 강함(결정적 이유).** 로드맵에 추출 정교화·"2개 이상 문서 공유 개념만" 필터·정규화 개선·개념 별칭이 줄줄이 있고, 이들은 전부 *그래프를 다시 만드는* 변경이다. A는 저장된 *문서*로 다시 빌드하면 개선이 **옛 데이터에 자동 반영**되어 마이그레이션 세금이 0이다. B는 빌더를 손볼 때마다 저장된 그래프가 옛 로직 결과로 굳어 있어 **데이터 마이그레이션을 반복**해야 한다.
- **진실의 원천 단일화.** 그래프가 파생이므로 원천은 문서다. A는 원천만 저장해 동기화할 게 없다. B는 원천(현 문서 상태)과 파생(저장 그래프)이 둘로 갈려 어긋남(동기화 버그)이 끼어들고, 문서 삭제 시 "이 노드가 어느 문서 것이었나" 역추적이 번거롭다.
- **빌드 비용은 무시 가능.** `buildGraph`는 메모리 내 소규모 연산이라 복원 시 재빌드 부담이 없고, 저장 용량도 작다(개념 텍스트 배열 수준).
- **포기한 것**: 새 `documents` 테이블·RPC·매핑을 만드는 *일회성* 작업. 단 [0011]의 원자적 교체 패턴(`replace_graph`)을 `replace_documents`로 거의 그대로 재사용해 비용을 줄인다. 옛 `nodes`/`edges` 테이블은 드롭(또는 보류).

## 결과 (이 결정으로 생기는 후속 영향·제약)

- **스키마**: `documents(id text, user_id uuid FK, label text, concepts text[], created_at)` + RLS(`user_id = auth.uid()`). 저장 = 사용자별 문서 전체 교체 RPC `replace_documents`(원자적).
- **코드**: `storage.ts` = `saveDocuments`/`loadDocuments`(옛 `saveGraph`/`loadGraph` 대체). `graph-rows.ts` = `rowsToDocuments`(DB 행 ↔ `DocumentInput[]`, **TDD 대상**).
- **마이그레이션**: dev/prod 분리([0012]) 위에서 — 로컬 `supabase start` 먼저, `db reset` 뒤 재로그인. (plan `2026-06-12-bipartite-graph.md` Task 8.)
- **면접 포인트**: "파생 결과가 아니라 원천만 저장해, 빌더 로직이 바뀌어도 저장 데이터가 안 굳게 했다"는 아키텍처 근거.
