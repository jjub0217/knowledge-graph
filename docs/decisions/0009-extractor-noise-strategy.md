# 0009. 추출 후보 노이즈 대응 전략 — 펜스 제외는 확정, 개념 판별 필터는 보류

- 상태: 채택
- 날짜: 2026-06-04

## 맥락 (왜 이 결정이 필요한가)
추출기([Task 3](../plans/2026-05-29-knowledge-graph-mvp.md))를 실제 학습 문서(`bug-fix.md`)에 돌려보니 후보가 **151개**나 나왔다. 화면이 후보로 가득 차 쓰기 어려웠다. "추출기가 엉망"이라 단정하기 전에 실제 문서로 출처별 분류(측정)를 했더니, 노이즈가 **성격이 다른 두 갈래**로 갈렸다. 어디까지·어떻게 줄일지 정할 필요가 있다.

## 결정
- **펜스 코드블록(``` … ```)은 추출 전에 제거한다(확정).** 코드 덩어리는 학습 "개념"이 아니므로 후보에서 뺀다. ([이슈 #10](https://github.com/jjub0217/knowledge-graph/issues/10) · PR #11에서 구현, TDD)
- **문장형 제목·파일 경로·코드 식별자 등 "개념이 아닌 정상 추출"은 자동 필터로 거르지 않고 보류한다.** 무엇이 "개념"인지 일반 규칙으로 정의하기 어렵고, 사람이 후보를 보고 채택하는 단계([ADR 0002](0002-node-source.md)의 human-in-the-loop)가 이미 거름망 역할을 한다.
- 추출기 수정은 **TDD**(테스트 먼저)로 한다.

## 고려한 선택지 / 무엇을 얻고 무엇을 포기했나
측정 결과 노이즈는 두 종류였다.

- **① 펜스 코드블록이 통째로 한 후보로 잡힘 (버그)** — 인라인 코드 정규식(`` /`([^`]+)`/g ``)이 세 겹 백틱 블록을 탐욕적으로 삼켜, `bash\ngit checkout …` 같은 여러 줄 코드가 한 후보로 들어왔다. **누가 봐도 버그** → 추출 전에 `/```[\s\S]*?```/g`로 펜스 블록을 제거. (얻음: 멀티라인 노이즈 제거 / 포기: 없음 — 코드 블록은 개념이 아님)
- **② 문장형 제목·경로·식별자 (개념이 아닌 정상 추출)** — `Step 5: Worktree에서 …` 같은 문장형 제목, `docs/conventions.md`·`SI-006` 같은 경로·코드. 길이 기준·문장 여부·경로 패턴 같은 **휴리스틱(어림짐작 규칙)으로 거를 수는 있으나**, "무엇이 개념인가"를 자동으로 판별하면 멀쩡한 후보까지 잘못 버릴 위험이 크다. → **보류.** (얻음: 멀쩡한 후보 보존·구현 단순 / 포기: 일부 비개념 후보가 목록에 남음 — 사람이 채택 단계에서 무시)

①만 고치고 ②를 분리한 이유: ①은 **고장**이고 ②는 **제품 판단**이다. 섣불리 ②까지 자동화하면 "잘 깨지는" 규칙이 되어 정작 필요한 개념을 놓친다.

## 결과 (이 결정으로 생기는 영향)
- 펜스 제거로 후보 **151 → 111(−26%)**, 40자 넘는 긴 노이즈는 다수 → **3**, 제목(heading)도 **40 → 24**(코드 블록 *안*에 있던 예시 제목까지 함께 사라진 덤).
- TDD로 회귀를 막았다 — 펜스 *밖* 멀쩡한 인라인 코드는 살리고 펜스만 제거됨을 테스트가 보장(`extractor.test.ts`).
- 코드리뷰 봇(Kimi)이 "중복 후보 key/ID 충돌"을 지적했으나, 추출기가 소문자 기준으로 중복을 이미 제거하므로 실제로는 일어나지 않는다 — diff만 본 **헛짚음**으로 판정(실측으로 반박).
- **2단계(개념 판별 필터)는 필요해지면 별도 이슈로 재검토.** 재검토 조건: 사람이 후보를 추리는 부담이 과도해질 때.
