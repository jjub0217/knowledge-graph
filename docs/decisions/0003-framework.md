# 0003. 앱을 만들 도구(프레임워크) = Next.js

- 상태: 채택
- 날짜: 2026-05-28

## 맥락 (왜 이 결정이 필요한가)
이 앱을 어떤 도구(프레임워크 — 웹앱을 만드는 뼈대 도구)로 만들지 정해야 한다. 결정에 영향을 준 제약 세 가지:
1. velog 블로그 글을 가져오려면 **작은 서버 한 조각**이 꼭 필요하다 — 브라우저가 velog를 직접 부르면 CORS(다른 사이트를 브라우저에서 직접 못 받게 막는 보안 규칙)에 막히기 때문([0004](0004-velog-fetch.md)에서 실측 확인).
2. 사용자가 가장 익숙한 스택(주력 도구)이어야 빨리 만들고 면접에서 설명도 쉽다.
3. 나중에 인터넷에 올려(배포) 포트폴리오로 보여줄 수 있어야 한다.

## 고려한 선택지
- A. **Next.js** — React 기반 프레임워크. 좋은 점: 서버에서 도는 코드(서버 라우트/서버 액션)가 기본 내장이라 velog용 "서버 한 조각"이 거의 공짜, 사용자 주력 스택, Vercel로 배포 한 방. 아쉬운 점: 순수 클라이언트 도구치곤 살짝 무거움.
- B. **Vite + React** — 가볍고 빠른 순수 브라우저 앱 빌드 도구. 좋은 점: 가벼움. 아쉬운 점: 서버가 없어서 velog CORS를 우회하려면 서버/프록시(중간에서 대신 받아주는 다리)를 따로 붙여야 함 → 결국 일이 늘어 이점이 사라짐.
- C. **순수 정적 HTML/JS** — 가장 가벼움. 아쉬운 점: 규모가 커지면 관리가 힘들고, 서버가 없어 velog를 못 가져옴.

## 결정
**A. Next.js.**

## 근거 / 무엇을 얻고 무엇을 포기했나
- velog를 가져오려면 서버 한 조각이 **반드시** 필요한데, Next.js는 그게 기본 내장이라 별도 설치·배치 없이 바로 쓴다. Vite(B)는 가볍지만 결국 서버를 따로 붙여야 해서 "가벼움"이라는 장점이 깎인다.
- 사용자 주력 스택이라 개발이 빠르고, 면접에서 구조를 설명하기 쉽다.
- 배포(Vercel)가 쉬워 포트폴리오로 올리기 좋다.
- 얻은 것: 서버 라우트 공짜 + 익숙함 + 쉬운 배포. 포기한 것: 순수 클라이언트 앱의 가벼움(약간의 무게를 감수).

## 결과 (이 결정으로 생기는 영향)
- velog 가져오기는 Next.js **서버 라우트**(서버에서 도는 코드 한 조각)에서 처리해 CORS를 우회한다([0004](0004-velog-fetch.md)).
- 그래프 그리기·점/선 편집은 **브라우저에서 도는 화면(클라이언트 컴포넌트)** 으로 만든다.
- 저장은 MVP에선 **브라우저 안 저장 공간(localStorage/IndexedDB)** + JSON 내보내기/가져오기. 데이터베이스(DB)는 필요해지면 별도 결정.
- 그래프를 실제로 그리는 라이브러리 선택은 다음 ADR(0005 후보)에서 따로 정한다.
