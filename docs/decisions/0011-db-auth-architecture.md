# 0011. DB 저장 + 인증 아키텍처 (Supabase 하이브리드)

- 상태: 채택
- 날짜: 2026-06-08

## 맥락 (Context)

[ADR 0010](0010-db-auth-priority.md)에서 DB 저장·인증을 로드맵 2번으로 앞당겼고, "스택·인증 방식 등 별도 설계가 필요하며 착수 시 새 ADR이 필요할 수 있다"고 남겼다. 이 문서가 그 후속으로, **무엇을 어떤 부품으로 만들지**를 결정한다.

전제: 주동기는 **포트폴리오 임팩트**(백엔드·인증·DB 역량을 보여주기) + 멀티기기 + 데이터 안전. 현재 저장은 localStorage([ADR 0003](0003-framework.md))이며 교체 지점은 `storage.ts`다. 데이터 모델은 노드·엣지·사용자로, 표(table)에 자연스럽게 들어맞는다.

## 고려한 선택지 (Options)

설계는 서로 얽힌 다섯 결정으로 나뉜다. 각 결정의 대안과 채택안은 아래와 같다.

### (1) 백엔드 스택
- **A. Supabase 직접(BaaS)** — 브라우저가 Supabase를 직접 호출, 권한은 RLS. 가장 빠르고 가볍지만 "서버를 직접 짠" 신호가 약하다.
- **B. 직접 백엔드** — Next.js 라우트 + ORM + 호스팅 Postgres + 인증 라이브러리를 모두 직접. 포폴 신호·학습 최대지만 가장 무겁고 보안 책임이 크다.
- **C. 하이브리드** — Supabase를 DB·인증으로 쓰되, 데이터 접근은 직접 쓴 Next.js API 라우트로 감싼다. 서버 코드도 있고 관리형 DB·인증도 활용한다.

### (2) DB 접근 방법
- **㉠ supabase-js** — Supabase 전용 클라이언트로 표를 다룬다. 추가 부품 없이 가볍다.
- **㉡ ORM(Drizzle 등)** — Postgres에 ORM을 따로 연결해 타입 안전 쿼리·마이그레이션 관리. 신호는 +α지만 부품·설정이 늘어 무겁다.

### (3) DB 스키마
- **JSON 통째** — 사용자 1명 = 1행, 그래프를 JSON 한 덩어리로. `storage.ts`와 거의 1:1이라 간단하지만 관계형 DB를 "파일 칸"처럼 써 설계 신호가 약하다.
- **정규화** — `nodes`·`edges` 표로 쪼개고 각 행에 `user_id`(FK). 스키마 설계 신호가 세고 FK·조인·트랜잭션을 다루지만, 저장이 통째 교체/upsert가 되어 더 무겁다.

### (4) 비로그인 방문자 / localStorage
- **유지** — 비로그인은 localStorage 그래프(예시·편집 영속), 로그인은 서버 DB. 저장 경로가 둘이라 분기가 필요하다.
- **제거(로그인 필수 저장)** — 비로그인은 예시 고정값만 보고 저장은 안 됨(새로고침 시 리셋), 저장 경로는 서버 DB 하나. 단순하지만 비로그인 편집은 영속되지 않는다.

### (5) 인증 세션 저장 위치
- **localStorage 토큰** — 클라이언트 전용 앱에 흔하다. 다만 서버 라우트가 세션을 읽으려면 요청마다 토큰을 수동으로 실어야 한다(서버는 localStorage를 못 읽는다).
- **쿠키 토큰(`@supabase/ssr`)** — 요청에 자동으로 실려 가 서버 라우트가 읽을 수 있다. `httpOnly`로 XSS에 강하다.

### (6) 소셜 로그인 제공자
- 후보: 일반 회원가입(이메일/비밀번호), Google, Kakao, Naver, Facebook.
- Naver는 Supabase 기본 미지원(커스텀 OAuth 필요), Facebook은 Meta 앱 심사 부담이 있다.

## 결정 (Decision)

- **(1) 스택 = C. 하이브리드** — Supabase(DB·인증) + 직접 쓴 Next.js `/api/graph` 라우트.
- **(2) DB 접근 = ㉠ supabase-js** — 별도 ORM 없음.
- **(3) 스키마 = 정규화** — `nodes`·`edges` 표 + `user_id`(FK) + RLS.
- **(4) localStorage = 제거(로그인 필수 저장)** — 비로그인은 예시 고정값만 표시, 저장은 로그인 사용자만.
- **(5) 세션 = 쿠키(`@supabase/ssr`)**.
- **(6) 제공자 = 1차 Google + Kakao**, 2차 이메일 회원가입. **Naver·Facebook 제외**.

## 근거 / 트레이드오프 (Rationale)

- **하이브리드(C)**: 주동기가 포폴 임팩트라 "서버 코드를 직접 짰다"는 신호가 필요하지만, 비밀번호 보안·관리형 Postgres 같은 어렵고 위험한 부분은 Supabase에 맡겨 위험·작업량을 줄인다. 풀스택 신호와 안전을 함께 얻는 중간 지점.
- **supabase-js(㉠)**: DB 설계 신호는 이미 정규화 스키마로 확보되므로, ORM까지 더하면 과설계로 기운다. ORM은 추후 "직접 백엔드(B)"로 더 갈 때 붙일 수 있다.
- **정규화**: "제대로 스키마를 설계했다"는 면접 스토리와 FK·조인·트랜잭션 학습을 얻는다. 저장 로직이 통째 교체로 무거워지는 비용은 그래프 규모가 작아 감당 가능하다.
- **localStorage 제거**: 예시 그래프는 모두에게 같은 **고정값**이라 사람별 저장이 필요 없고(코드 상수로 충분), 비로그인 편집의 영속화는 "로그인해야 저장"이라는 자연스러운 패턴으로 대체된다. 그 결과 저장 경로가 서버 DB 하나로 단순해진다. 포기한 것: 비로그인 편집이 새로고침 후에도 남던 편의.
- **쿠키 세션**: 하이브리드는 서버 라우트가 세션을 읽어야 하는데, 서버는 localStorage를 못 읽는다. 쿠키는 요청에 자동으로 실려 가 이 구조에 맞고, `httpOnly`로 보안 이점도 있다. (클라이언트 전용 앱이면 localStorage 토큰이 흔하지만, 이 앱은 서버를 거치므로 갈린다.)
- **제공자 범위**: 포폴에서 점수를 주는 것은 제공자 개수가 아니라 "인증 흐름을 제대로 설계했다"는 신호다. Naver는 미지원이라 작업이 몇 배인데 효용은 버튼 하나뿐이고, Facebook은 심사 부담이 크다 — 둘 다 효용 대비 비용이 안 맞아 제외. 이메일 회원가입은 가치가 있으나 인증 메일·비밀번호 재설정 흐름이 딸려와 2차로 분리한다.

## 결과 (Consequences)

- 저장 계층(`storage.ts`)이 **동기 → 비동기(async)** 로 바뀌고, 로그인 여부로 경로가 갈린다. `hasStoredGraph()`와 localStorage 그래프 저장은 제거된다(`exportJSON`/`importJSON`은 유지).
- 새 부품: Supabase 프로젝트(DB + RLS), `/api/graph` 라우트, `/auth/callback` 라우트, supabase 클라이언트(브라우저·서버), 로그인/로그아웃 UI, 행↔그래프 변환 순수 함수.
- 변환 함수는 순수 함수라 **TDD 셋째 사이클** 대상이 된다. 인증·라우트·OAuth는 자동 테스트가 닿기 어려워 수동 검수로 검증한다.
- 시크릿(Supabase 키)·소셜 앱 키 발급·Vercel 환경변수 등록은 사용자가 직접 수행한다. service_role 키는 쓰지 않는다.
- 이 결정은 [ADR 0003](0003-framework.md)의 프레임워크 선택(Next.js)을 유지하며, MVP 저장으로서의 localStorage가 타당했다는 판단도 뒤집지 않는다. 바뀌는 것은 저장 계층의 구현뿐이다.
- 상세 청사진은 [설계 문서](../specs/2026-06-08-db-auth-design.md)에 있다.
