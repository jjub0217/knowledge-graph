# 0001. 노드를 얼마나 잘게 나눌까(입도) = 개념/키워드 단위

- 상태: 채택
- 날짜: 2026-05-28

## 맥락 (왜 이 결정이 필요한가)

이 프로젝트의 핵심 가치는 **관계 발견**이다 — 따로 배운 개념들이 사실은 연결돼 있다는 걸 그림으로 발견하는 것. 그래프에서 "무엇끼리 연결되는가"는 결국 **점(노드) 하나를 무엇으로 보느냐**로 정해진다. 점 하나를 너무 큰 덩어리로 잡으면(예: 'React' 통째로), 그 안의 작은 연결이 안 보인다. 점을 적당히 잘게 쪼개야(예: 'useEffect', '클로저') 개념들 사이에 숨어 있던 관계가 비로소 드러난다.

## 고려한 선택지

- A. **개념/키워드** ("React useEffect", "DB 인덱스") — 가장 잘게. 좋은 점: 개념 사이 숨은 연결을 보기에 가장 좋음(관계 발견에 유리). 아쉬운 점: 점 수가 많아질 수 있음.
- B. **학습 항목(기록 하나)** (질문답 1개, 블로그 글 1편) — 좋은 점: 로그 구조와 1:1로 단순. 아쉬운 점: '기록' 단위라 개념 사이가 아니라 글과 글 사이 연결이 되어 발견이 둔해짐.
- C. **주제/태그** ("React", "DB") — 좋은 점: 점 수가 적어 단순. 아쉬운 점: 세밀한 발견이 안 되고 큰 덩어리만 보임.
- D. **두 단계로 섞기(주제 점 + 그 아래 개념 점)** — 좋은 점: 주제별로 묶이는 게 한눈에 잘 보임. 아쉬운 점: 점 종류가 둘이라 데이터 구조와 화면이 복잡해짐.

## 결정

**A. 개념/키워드 단위로, 점은 한 종류만(평평한 구조)** 쓴다. 주제는 따로 점으로 만들지 않고, 개념 점의 **색깔**로 나타낸다.

## 근거 / 무엇을 얻고 무엇을 포기했나

- 숨은 연결(관계 발견)은 개념을 잘게 쪼갰을 때 제일 많이 나온다. 그러니 A(개념 단위)가 핵심 가치(관계 발견)에 **직접 도움이 된다**.
- **D(두 단계)의 진짜 매력은 "주제별로 무리 지어 보인다"는 점**이다. 그런데 그 장점은 **점 종류를 둘로 늘리지 않고도** 얻을 수 있다 — 개념 점에 `주제`를 **색깔**로 입히면 된다: _같은 주제 = 같은 색_ → 같은 색 덩어리가 곧 주제 무리. 즉 **D의 핵심 장점을 색깔로 가져오면서**, 데이터 구조와 화면은 한 종류짜리 단순한 구조로 유지해 복잡함을 피한다.
- 얻은 것: 세밀한 발견 + 단순한 데이터 구조와 그리기 로직.
- 포기한 것: 주제를 클릭하거나 접었다 폈다 하며 다루는 기능(주제를 따로 오르내리는 탐색). 지금은 필요 없으니 뺀다 — 필요해지면 그때 따로 결정.

## 결과 (이 결정으로 생기는 영향)

- 개념 점의 데이터 모양: `{ id, label(개념 이름), topic(주제), ... }` — `topic`이 색깔을 정하는 입력이 된다.
- 주제를 색깔에 연결하는 표가 필요하다(주제 ↔ 색).
- 개념 점이 많아지면 화면이 빽빽해질 수 있다 → **검색/거르기** 기능이 다음 과제로 필요해짐(다음 ADR(왜 이렇게 정했는지 이유를 적어두는 결정 기록) 후보).
- 엣지(연결선)·데이터 출처 관련 결정은 [0002](0002-node-source.md) 참조.
