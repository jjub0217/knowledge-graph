# 0002. 개념 점을 어디서 가져올까 = 자동 추천 + 사람 확인 (섞기)

- 상태: 일부 대체됨(→ [0013](0013-bipartite-model-pivot.md))
- 날짜: 2026-05-28

> ⚠️ 2026-06-12 [0013]으로 **"사람 확인(채택) + 손으로 연결"**이 대체됨 — 이제 개념 노드는 전부 자동(표기 정규화), 엣지도 문서→개념 자동 멤버십(수동 연결 제거). 단 "추출기는 후보를 뽑는 순수 함수(TDD 첫 연습거리)"라는 점은 그대로 유효.

## 맥락 (왜 이 결정이 필요한가)

점(노드)은 개념/키워드 단위로 정했다([0001](0001-node-granularity.md)). 이제 그 개념 점을 **어떻게 채울지**(어디서 가져올지)를 정해야 한다. 이 프로젝트의 핵심은 흩어진 **학습 기록**(`~/Desktop/weekly-learning-log.md` 등)을 활용하는 것이다. 한편 연결선(엣지, "관련 있음"을 잇는 방향 없는 선)은 MVP에서 **손으로 직접** 긋기로 했다 — 이 결정은 점을 채우는 방식에 관한 것이다.

## 고려한 선택지

"그래프에 들어갈 개념 점들을 어디서, 어떻게 만들어 내느냐" — 즉 점을 채우는 방법을 고르는 거예요.
네 개는 다 같은 목표(점 채우기)인데 누가 그 일을 하느냐가 다릅니다.

```
┌────────────────┬─────────────────┬───────────────────────────────────┐
│ 선택지           │ 누가 점을 만드나    │ 무슨 뜻                            │
├────────────────┼──────────────────┼──────────────────────────────────┤
│ A. 순수 손입력    │ 100% 사람         │ 내가 개념을 하나하나 직접 타이핑         │
├────────────────┼──────────────────┼──────────────────────────────────┤
│ B. 완전 자동     │ 100% 기계          │ 프로그램이 로그 읽고 알아서 점 다 생성    │
├────────────────┼───────────────────┼──────────────────────────────────┤
│ C. 섞기 (채택)   │ 기계 추천 + 사람 확정  │ 기계가 후보 제안 → 내가 고르고 고침      │
├────────────────┼───────────────────┼──────────────────────────────────┤
│ D. 형식 입력     │ 사람(미리 표시)       │ 로그에 개념을 미리 태그해두고 그대로 읽음  │
└────────────────┴───────────────────┴──────────────────────────────────┘
```

- A. **순수 손입력** — 좋은 점: 단순하고 확실. 아쉬운 점: 학습 기록을 못 살리고, 글에서 키워드를 뽑아내는 로직이 없어 TDD로 연습할 거리가 없음.
- B. **로그에서 완전 자동 추출** — 좋은 점: 기록 활용·자동화, TDD 연습거리 풍부. 아쉬운 점: 한국어에서 "의미 있는 학습 개념"만 골라내는 건 정확도 한계가 분명(말이 붙어 변하고 합쳐진 단어가 많고 앞뒤 맥락에 따라 뜻이 달라져서 → 엉뚱한 걸 뽑거나 중요한 걸 놓침).
- C. **섞기 (자동으로 추천 → 사람이 고르기/고치기 + 손으로 추가)** — 좋은 점: 활용·자동화와 정확도를 적절히 맞바꿈. 아쉬운 점: 사람이 확인하는 단계가 들어가고, 만들 게 A보다 조금 많음.
- D. **정해진 형식으로 입력(로그에 개념을 표시해두기)** — 좋은 점: 뽑아낼 필요 없이 정확. 아쉬운 점: 기록 습관을 바꿔야 함(글머리 표시·태그를 매번 달아야).

## 결정

**C. 섞기.** 로그를 읽어 **후보 키워드를 뽑아 추천**하고, 받아들일지·고칠지·버릴지 최종 판단은 **사람**이 한다(손으로 직접 추가도 가능).

## 근거 / 무엇을 얻고 무엇을 포기했나

- **글에서 키워드를 뽑아내는 로직(글 → 후보 키워드)은 순수 함수다**: 같은 글을 넣으면 늘 같은 결과가 나오고, 바깥 상태를 건드리지 않는다 → 가짜로 흉내 내는 준비(모킹) 없이 바로 테스트할 수 있다. 그래서 이 프로젝트의 **TDD 첫 연습거리로 딱 맞는다** — 실패하는 테스트 먼저(RED) → 통과하는 최소 코드(GREEN) → 정리(REFACTOR) 순으로, **테스트를 먼저** 짜고 추출기를 만든다.
- 하지만 한국어에서 학습 개념을 자동으로 뽑는 건 **정확도 한계가 분명**하다(엉뚱한 걸 뽑거나 놓치는 게 어쩔 수 없음). 완전 자동(B)은 잘못된 점으로 그래프를 지저분하게 만들어 핵심 가치인 '관계 발견'을 해친다.
- 그래서 **사람이 중간에 확인하는 방식**을 쓴다: 기계는 후보를 _추천만_ 하고, 받아들일지는 사람이 정한다. 이렇게 하면 (1) 정확도는 사람이 책임지고 (2) 활용·자동화는 기계가 맡고 (3) **추출기가 완벽하지 않아도 되게** 만든다(완벽할 필요 없이 "사람이 검토하는 부담만 줄이면" 됨).
- 얻은 것: 학습 기록 활용 + TDD 연습거리 + 데이터 품질(사람이 보증).
- 포기한 것: 완전 자동의 "클릭 없이 끝" 편함(추가할 때마다 사람이 확인하는 비용), D의 정확함(대신 기록 습관을 바꾸는 부담은 피함).

## 결과 (이 결정으로 생기는 영향)

- 처리 흐름: `로그 읽기 → 후보 뽑기(순수 함수, TDD) → [사람 확인 화면] → 점 확정`.
- 추출기는 후보뿐 아니라 (가능하면) **그 후보가 글의 어디에서 나왔는지 위치**도 함께 돌려줘 사람이 판단하기 쉽게 한다.
- **후보 확인 화면**(후보 목록 + 받기/고치기/버리기)이 MVP 필수 요소로 들어간다.
- 추출 정확도의 목표는 "완벽"이 아니라 **"사람이 검토하는 부담을 최소로"** — 재고 다듬는 것도 이 기준으로.
- **글 출처는 두 곳**: ① 학습 로그(파일 올리기) ② velog 블로그 글(주소 붙여넣어 가져오기). 둘 다 같은 "섞기" 흐름(뽑기 → 사람 확인)을 그대로 탄다. velog를 어떻게 가져오는지는 [0004](0004-velog-fetch.md) 참조.
