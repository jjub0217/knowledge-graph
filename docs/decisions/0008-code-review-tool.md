# 0008. 코드리뷰 도구 — Kimi(K2.6) · OpenRouter 경유 · 커스텀 워크플로

- 상태: 채택
- 날짜: 2026-06-01

## 맥락 (왜 이 결정이 필요한가)
GitHub 공개([ADR 0007](0007-workflow-conventions.md)) 후, PR마다 AI 코드리뷰를 자동화하기로 했다. 정할 것은 ① 어떤 모델 ② 어떤 접근 경로 ③ 어떤 전달 방식이다. cuddle-market도 같은 결정을 내렸지만 **"왜 OpenRouter인지"가 어디에도 기록되지 않아** 사후에 이유를 복구할 수 없었다(주간 로그·세션 히스토리 검색해도 없음). 이 ADR은 그 전철을 밟지 않으려 근거를 남긴다.

## 결정
- **모델 = Kimi K2.6** (Moonshot AI). 레거시 K2는 2026-05-25 EOL이라 제외, K2.5보다 코딩 워크로드에 강한 최신.
- **접근 = OpenRouter 경유**(여러 모델을 한 키로 중계하는 애그리게이터). 기존 OpenRouter 키 재사용.
- **전달 = 커스텀 GitHub Actions 워크플로**(`.github/workflows/kimi-review.yml`). PR diff를 OpenRouter로 보내 한국어·실용 프롬프트로 리뷰받아 PR 댓글로 단다.
- **적용 시점 = UI(Task 7) PR부터.** 토대(Task 1~6)는 단일 PR로 머지하며 리뷰 생략. 머지는 사용자가 직접.

## 고려한 선택지 / 무엇을 얻고 무엇을 포기했나
- **모델: Kimi vs Claude** — Claude는 리뷰 품질이 좋으나 PR당 비용 부담($0.10~0.50, cuddle-market 경험). Kimi가 훨씬 저렴 → **비용**으로 Kimi 채택. (포기: Claude의 리뷰 정밀도)
- **접근: OpenRouter vs Moonshot 직접** — 직접은 prefix 캐싱 등 제조사 혜택 가능성이 있으나, **기존 키 재사용 + 한 키로 여러 모델**의 편의를 우선해 OpenRouter. (재검토 조건: 리뷰 비용이 커지면 direct+캐싱과 비교)
- **전달: 커스텀 vs 마켓플레이스 액션** — 액션은 인라인 주석·셋업 적음이 장점이나, **시스템 프롬프트를 프로젝트 철학("과도한 방어코딩 지적 말기" 등)대로 제어**하려고 커스텀 선택. (포기: 인라인 주석)

## 결과 (이 결정으로 생기는 영향)
- 워크플로가 PR(opened·synchronize)에 자동 실행, **`secrets.OPENROUTER_API_KEY`** 사용 — 키를 코드·워크플로에 직접 노출하지 않음.
- 봇은 **diff만** 보므로 거짓 양성이 날 수 있다 → 지적은 사람이 진위 판단 후 반영(리뷰는 강제 게이트가 아니라 참고).
- OpenRouter의 Kimi K2.6 모델 슬러그(`moonshotai/kimi-k2.6`)는 **첫 실행에서 확인** — 슬러그가 다르면 워크플로에서 수정.
- 의사결정 과정은 기술 블로그 글("AI 코드리뷰 도구, 무엇으로 고를까")로도 정리.
