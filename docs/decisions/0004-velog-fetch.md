# 0004. velog 블로그 글 가져오기 = GraphQL 본문 + 서버 라우트

- 상태: 채택
- 날짜: 2026-05-28

## 맥락 (왜 이 결정이 필요한가)
블로그(velog) 글도 개념 점의 출처로 쓰기로 했다([0002](0002-node-source.md)). 그런데 velog는 "이렇게 가져가세요"라고 공개한 정식 창구(공식 REST API 문서)가 없다. 그래서 글 본문(마크다운)을 **어떻게 가져올지**, 그리고 브라우저에서 직접 부르면 막히는 **CORS**(다른 사이트를 브라우저에서 직접 못 받게 막는 보안 규칙) 문제를 어떻게 풀지 정해야 한다.

## 고려한 선택지
- A. **GraphQL로 본문 받기** — velog 내부의 GraphQL(서버에 "이 글 본문 줘"라고 정해진 방식으로 묻는 창구) `https://v2.velog.io/graphql`에 `post(username, url_slug){ body }`를 물어본다. 좋은 점: **원본 마크다운을 거의 그대로** 받음(실측 확인). 아쉬운 점: 비공식 창구라 velog가 바꾸면 깨질 수 있고, 가끔 본문에 `<table>` 같은 HTML이 섞임.
- B. **페이지 HTML 받아서 파싱** — 글 주소를 그냥 받으면 마크다운이 아니라 변환된 HTML이 온다. 아쉬운 점: 지저분하고 화면 구조가 바뀌면 잘 깨짐.
- C. **RSS**(블로그 글 목록을 기계가 읽기 좋게 뿌려주는 표준) — `https://v2.velog.io/rss/@사용자명`. 좋은 점: 글 목록(제목·주소)을 받기 좋음. 아쉬운 점: 본문은 최근 몇 개·HTML형이라 "본문 소스"로는 부족.

## 결정
- 본문은 **A(GraphQL `body`)** 로 가져온다.
- 글 목록 불러오기(여러 글 한 번에)에는 **C(RSS)** 를 보조로 쓴다.
- CORS는 **Next.js 서버 라우트**가 velog를 대신 호출해 우회한다([0003](0003-framework.md)).

## 근거 / 무엇을 얻고 무엇을 포기했나
- 2026-05-28 **실측 결과**: velog의 GraphQL 창구(`v2.velog.io/graphql`)에 글의 본문(body)을 요청하면 원본 마크다운(`## 정적(static)`, 코드블록 ` ```ts ` 까지)이 깨끗하게 돌아온다. HTML 파싱(B)보다 깨끗하고 안정적이라 A로 결정한다.
- 같은 실측에서 응답에 `access-control-allow-origin`(브라우저 직접 호출 허용 표시) **헤더가 없었다** → 브라우저가 직접 부르면 막힌다. 그래서 서버 라우트가 대신 부른다.
- 얻은 것: 깨끗한 마크다운 → 키워드 뽑기 품질이 좋고, 로그와 **같은 "섞기" 흐름**을 그대로 재사용. 포기한 것: 비공식 창구 의존(언제든 깨질 수 있음).

## 결과 (이 결정으로 생기는 영향)
- 서버 라우트: velog 글 주소(또는 사용자명+slug)를 받아 GraphQL을 호출하고 마크다운을 돌려준다. 주소에서 사용자명·slug를 뽑아내는 처리가 필요.
- 비공식 창구라 깨질 위험을 한 군데(서버 라우트)에 가둬 영향 최소화 + **에러 처리**를 둔다. 안 되면 **"마크다운 직접 붙여넣기"** 대체 경로로 빠질 수 있게 한다.
- 본문에 섞인 inline HTML(`<table>` 등)은 키워드 뽑개가 무시하거나 글자만 취한다.
- **우리 앱은 GraphQL 라이브러리를 도입하지 않는다** — velog의 GraphQL 창구에 `fetch`로 요청 한 번(POST + JSON 본문) 보낼 뿐이다. GraphQL은 velog 쪽 데이터 창구 방식이지 이 앱이 채택한 기술 스택이 아니다.
