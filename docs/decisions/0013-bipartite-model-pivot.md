# 0013. 이분 그래프 모델로 피벗 = 문서·개념 두 노드 + 자동 멤버십 엣지

- 상태: 채택
- 날짜: 2026-06-12
- 관련: [0001](0001-node-granularity.md) 일부 대체 · [0002](0002-node-source.md) 일부 대체 · [0009](0009-extractor-noise-strategy.md) · 로드맵([roadmap.md](../roadmap.md))

> 쉬운 한국어로 작성. 어려운 말은 뒤에 괄호로 풀이.
> 용어: **이분 그래프**(노드가 두 *종류*로 갈리고, 엣지는 *서로 다른 종류 사이*에만 잇는 그래프), **멤버십**(소속 = "이 문서가 이 개념을 담고 있다"는 포함 관계), **degree**(한 노드에 붙은 연결선 수).

## 맥락 (왜 이 결정이 필요한가)

이 프로젝트가 스스로 내건 핵심 가치는 **관계 발견** — "따로 배운 개념·자료가 *알고 보니* 이어져 있더라"를 눈으로 깨닫는 것이다. 그런데 현재 MVP(ADR [0001]·[0002] 기반)는 **엣지(연결선)를 사람이 손으로 긋는다**. 손으로 긋는다는 건 *내가 이미 아는 연결만* 그릴 수 있다는 뜻이고, **내가 몰랐던 연결은 영원히 안 나타난다.** 즉 비전("발견")과 구현("수동 그리기")이 정반대로 어긋나 있었다.

원래 사용자의 의도는 "흩어진 내 학습 `.md`·기술 블로그 글들이 *서로* 연관돼 있다는 걸 보여주는 것"이었다. 이를 살리려면 연결을 손으로 긋는 대신 **자료의 내용에서 자동으로 끌어내야** 한다. 비교 대상인 Obsidian조차 이걸 "분석"으로 하지 않는다 — Obsidian의 `[[링크]]`는 사람이 노트를 쓰며 직접 타이핑한 명시적 링크를 기계적으로 그릴 뿐이다(추론 없음). 우리는 `[[ ]]`를 안 써도 연결이 뜨게 하려 한다. 그것이 차별점이다.

## 고려한 선택지

연결을 자동화하려면 **"점을 무엇으로 보고, 엣지를 무엇으로 정의하느냐"**부터 다시 정해야 한다.

- **A. 점 = 개념만, 엣지 = 개념끼리 자동 관계.**
  "클로저와 렌더링이 관련 있나?"를 기계가 판단해야 함 → 의미 판단이 필요해 어렵고(로드맵 7에서 "가장 어려워 맨 뒤"로 미뤘던 바로 그 문제), LLM·임베딩을 쓰면 **비결정적**이 되어 "왜 이 둘이 이어졌나"를 설명하기 어렵다(메타목표 = 설명력에 역행).
- **B. 점 = 문서/글만, 엣지 = 두 문서의 공유 개념 수.**
  "문서끼리 연결"은 직접적이나, *무엇을 통해* 이어지는지(다리 개념)가 안 보이고, 개념 자체가 노드가 아니라 후속 능동 학습(개념 클릭→퀴즈)의 토대가 약하다.
- **C. 점 = 문서 + 개념 둘 다(이분 그래프), 엣지 = 문서→개념(멤버십)만.**
  "개념 A와 B가 관련 있나"라는 어려운 판단을 **아예 안 한다.** 두 문서는 같은 개념 노드를 가리키므로 그 개념을 *다리* 삼아 자동으로 이어진다. 의미 판단 없이 "누가 무엇을 담았나"라는 사실만 기록.
- **D. (보조 질문) 개념 동일성·관계 판단을 LLM에 위임.**
  가장 강력하나 API·비용·비결정성을 동반하고, C를 택하면 *관계 판단 문제 자체가 사라지므로* 지금은 오버엔지니어링. LLM의 제자리는 추출이 아니라 **로드맵의 퀴즈 단계**(자유 답안 채점 = LLM만 가능, 그래프는 결정적 유지)다.

## 결정

**C. 이분 그래프로 피벗한다.** 구체적으로:

1. **노드 두 종류** — `document`(올린 `.md`·velog 글)와 `concept`(거기서 뽑은 개념). (← [0001]의 "점은 한 종류만(평평한 구조)"을 대체.)
2. **엣지 = 문서→개념 멤버십, 전부 자동.** "점 두 번 클릭 = 수동 연결" 기능은 **제거.** (← [0002]의 "손으로 연결" 전제를 대체.)
3. **개념 노드 = 추출된 것 전부 자동 + 표기 정규화.** 사람이 하나씩 채택하는 단계 없이, 드롭하면 그래프가 저절로 뜬다. (← [0002]의 "사람 확인" 전제를 대체.)
4. **개념 동일성 = 표기 정규화①만**(소문자·공백·구분기호 통일 → 같은 표기 = 같은 개념 노드). 별칭 사전②·형태소·임베딩·LLM은 **보류**(후속 ADR 후보).
5. **다중 입력 누적** — 여러 자료를 *쌓아* 한 그래프에. (자료 1개만으론 자료 간 연결이 안 보이므로 C의 전제.)
6. **LLM의 자리 = 로드맵 퀴즈 단계.** 추출·연결 파이프라인에는 LLM을 넣지 않는다(그래프를 결정적·설명 가능하게 유지).

## 근거 / 무엇을 얻고 무엇을 포기했나

- **얻은 것 — 어려운 문제를 쉬운 문제로 치환.** C는 "관계 판단(어려움·비결정)"을 "공유 멤버십(쉬움·결정적)"으로 바꾼다. 덕분에 파이프라인 전체가 **결정적이고 설명 가능**해져, 이 프로젝트의 1급 관심사(면접에서 "어떻게 만들었는지" 설명)에 직접 부합한다.
- **얻은 것 — 진짜 발견 + Obsidian 차별점.** 사람이 안 그어도 연결이 뜨므로 "몰랐던 연결"이 보인다. Obsidian은 `[[ ]]`를 미리 쳐야 하지만 우리는 안 친다.
- **얻은 것 — 후속 성장과의 정합성.** 개념이 1급 노드라 로드맵의 퀴즈(개념 클릭→측정)·이해도 표시·추천이 자연스럽게 얹힌다. "클로저 약하네 → 클로저를 다룬 네 자료 여기"가 그래프 구조에서 공짜로 나온다.
- **포기한 것 — 의미적 동일성.** "클로저"와 "closure", "useEffect"와 "use effect"는 표기가 달라 아직 다른 노드다. 정규화①로 표기 차이 일부만 흡수한다. (별칭②·임베딩은 후속.)
- **포기한 것 — 사람의 채택 통제감.** 전부 자동이라 노이즈가 그래프에 바로 올라온다. 추출기가 순진하므로([0009]) 노이즈가 거슬리면 다음 단계에서 "2개 이상 문서가 공유한 개념만 노드" 또는 추출 정교화(로드맵)로 좁힌다.
- **포기한 것 — 단일 자료의 풍부함.** 자료를 1개만 넣으면 "자료 간 연결"이 안 생겨 가치가 약하다. 여러 개를 쌓는 사용이 전제다.

## 결과 (이 결정으로 생기는 후속 영향·제약)

- **데이터 모델 변경**: 노드에 `kind: 'document' | 'concept'` 추가. 엣지는 `document_id → concept_id` 멤버십. (옛 `{id,label,topic}` 평평 구조 대체 — 새 spec에서 확정.)
- **코드 영향**(요약 — 상세는 새 spec):
  - 재사용: `extractor`, 저장/인증(Supabase), `react-force-graph` 렌더, 검색/필터.
  - 손봄: 데이터 모델·타입, `InputPanel`(한 건 교체 → 다중 누적 + 문서 노드 생성), 엣지 생성(수동 클릭 → 문서-개념 자동), `GraphView`(노드 종류별 모양·색), `graph-ops`(고립 = degree 0 의미 재정의), `example-graph`(이분 예시로).
  - 삭제: 수동 엣지(점 2번 클릭) 로직.
  - 신규: 표기 정규화 함수(순수 함수, TDD 대상).
- **고립 강조 의미 변화**: 이제 degree 0 = "아무 개념도 공유 안 한 자료" 또는 "한 문서에만 나온 개념" → "아직 네 지식망과 안 엮임" 힌트로 재해석.
- **문서 갱신 필요**: `README.md`(MVP 표·로드맵이 스테일), spec(신규 작성, 옛 spec은 기록 보존), `roadmap.md`(7번 흡수·재배치), `CLAUDE.md`, `HANDOFF.md`.
- **[0001]·[0002] 상태**: "일부 대체됨(→0013)"로 표기(삭제하지 않음 — 피벗 이력 자체가 면접 스토리).
- **후속 ADR 후보**: 개념 별칭/동의어②, 임베딩 기반 의미 유사도, 퀴즈에서의 LLM 연동 방식(가벼움 vs 무거움).
