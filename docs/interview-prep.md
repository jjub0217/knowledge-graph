### Q. 프레임워크는 왜 Next.js예요?
- **말하기 좋은 버전**:
  > "velog 블로그 글을 가져오려면 서버가 한 조각 필요했어요. 브라우저에서 velog를 직접 부르면 CORS(다른 사이트를 브라우저에서 직접 못 받게 막는 보안 규칙)에 막히거든요. Next.js는 서버 라우트가 기본 내장이라 그 조각이 거의 공짜였고, 제 주력 스택이라 빠르게 만들고 Vercel 배포도 쉬워서 골랐어요."
- 더 물으면: 순수 클라이언트면 Vite가 더 가볍지만 결국 CORS용 서버를 따로 붙여야 해 이점이 사라진다 / 근거는 ADR [0003](decisions/0003-framework.md).

### Q. 개념 점을 채우는 방식, 왜 "섞기(C)"로 했어요?
- **말하기 좋은 버전**:
  > "개념 점을 채우는 방법으로 네 가지를 비교했어요 — ① 손으로 직접 입력, ② 프로그램이 로그에서 전부 자동으로 뽑기, ③ 기계가 후보를 추천하고 사람이 확정하는 섞기, ④ 로그에 미리 형식을 정해 표시해두기. 완전 자동은 한국어라 엉뚱한 걸 뽑는 정확도 문제가 크고, 손입력은 애써 쌓은 학습 기록을 못 살려요. 그래서 기계는 후보만 추천하고 받아들일지는 사람이 정하는 '섞기'로 갔습니다 — 사람이 중간에서 확인하니 추출기가 완벽하지 않아도 됩니다."
- 더 물으면: 후보를 뽑는 로직은 순수 함수라 TDD(테스트 먼저)로 짰다 / 네 선택지의 자세한 맞바꿈은 ADR [0002](decisions/0002-node-source.md)의 `고려한 선택지`에.

### Q. velog 글을 어떻게 가져왔어요? (가장 까다로웠던 부분)
- **말하기 좋은 버전**:
  > "velog는 '이렇게 가져가세요' 하는 공개 API 문서가 없었어요. 그래서 직접 찔러보며 방법을 찾았는데, 내부 GraphQL에 글을 요청하니 원본 마크다운이 깨끗하게 오더라고요(HTML을 긁어 파싱하는 것보다 훨씬 안정적). 다만 브라우저에서 직접 부르면 CORS로 막혀서, Next.js 서버 라우트가 대신 호출하게 우회했어요. 비공식 창구라 깨질 수 있으니 한 군데(서버 라우트)에 가두고, 안 되면 마크다운을 직접 붙여넣는 대체 경로도 뒀습니다."
- 더 물으면: RSS·GraphQL·페이지 HTML 세 경로를 실측해 비교했다 / 근거는 ADR [0004](decisions/0004-velog-fetch.md).

## 프로세스 스토리 (차별화 포인트)
- 체계적 설계 프로세스: brainstorming → spec → plan → 구현 (superpowers 워크플로우).
- (TDD 도입 시) 테스트 우선 방식을 처음 적용해본 경험과 그 판단 근거.
- 아이디어를 후보 여러 개로 펼치고 트레이드오프로 좁힌 의사결정 과정.

### ★ 선행 규율 — "처음부터 게이트로 굴렸다"
> 코드를 *쓴 뒤* 품질을 붙인 게 아니라, **코드 첫 줄 전에** 규율을 강제로 깔고 시작했다는 스토리.

- **말하기 좋은 버전**:
  > "저는 코드 첫 줄을 쓰기 전에 타입체크·lint·테스트 게이트를 hook으로 강제하고 TDD로 시작했어요. 그래서 (커밋 N개) 내내 게이트가 깨진 적이 없습니다."
- **왜 차별화되나**: 대부분 "나중에 테스트 붙였어요"인데, 이건 "처음부터 규율로 굴렸어요". 품질이 *사후 보정*이 아니라 *기본값*.
- **근거(만들면서 채울 것)**:
  - 깐 게이트: tsc / lint / format / 테스트 러너 — (스택 정해지면 기입)
  - 강제 방식: hook(PreToolUse) 또는 pre-commit — CLAUDE.md "권유"가 아니라 hook "강제"로 한 이유 → `docs/decisions/`(ADR)에 기록
  - 증거: 게이트 통과 커밋 수 / 게이트가 실제로 막아준 사례 → `docs/journal/`에 기록
- **연결**: "약한 게이트(빌드/타입) vs 강한 게이트(테스트)"를 구분해 적용했다 (게이트 강도 인식).
