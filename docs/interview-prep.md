### Q. 프레임워크는 왜 Next.js예요?
- **말하기 좋은 버전**:
  > "velog 블로그 글을 가져오려면 서버가 한 조각 필요했어요. 브라우저에서 velog를 직접 부르면 CORS(다른 사이트를 브라우저에서 직접 못 받게 막는 보안 규칙)에 막히거든요. Next.js는 서버 라우트가 기본 내장이라 그 조각이 거의 공짜였고, 제 주력 스택이라 빠르게 만들고 Vercel 배포도 쉬워서 골랐어요."
- 더 물으면: 순수 클라이언트면 Vite가 더 가볍지만 결국 CORS용 서버를 따로 붙여야 해 이점이 사라진다 / 근거는 ADR [0003](decisions/0003-framework.md).

### Q. 개념 점을 채우는 방식, 왜 "섞기(C)"로 했어요?
- **말하기 좋은 버전**:
  > "개념 점을 채우는 방법으로 네 가지를 비교했어요 — ① 손으로 직접 입력, ② 프로그램이 로그에서 전부 자동으로 뽑기, ③ 기계가 후보를 추천하고 사람이 확정하는 섞기, ④ 로그에 미리 형식을 정해 표시해두기. 완전 자동은 한국어라 엉뚱한 걸 뽑는 정확도 문제가 크고, 손입력은 애써 쌓은 학습 기록을 못 살려요. 그래서 기계는 후보만 추천하고 받아들일지는 사람이 정하는 '섞기'로 갔습니다 — 사람이 중간에서 확인하니 추출기가 완벽하지 않아도 됩니다."
- 더 물으면: 후보를 뽑는 로직은 순수 함수라 TDD(테스트 먼저)로 짰다 / 네 선택지의 자세한 맞바꿈은 ADR [0002](decisions/0002-node-source.md)의 `고려한 선택지`에.

### Q. AI 코드리뷰는 어떤 도구로 붙였어요?
- **말하기 좋은 버전**:
  > "PR마다 AI 코드리뷰가 자동으로 돌게 했는데, 모델은 비용을 보고 골랐어요. Claude는 품질은 좋지만 PR당 비용이 부담이라 더 싼 Kimi로 갔습니다. Kimi를 부르는 길도 둘이라 — 여러 모델을 한 키로 중계하는 OpenRouter(편의점)냐, 제조사 직접(로스터)이냐 — 쓰던 키를 재사용할 수 있는 OpenRouter로 갔고요. 봇은 바뀐 부분(diff)만 보니 거짓 양성이 날 수 있어서, 지적을 다 고치지 않고 진짜 위반인지 사람이 한 번 거릅니다."
- 더 물으면: 도구를 6개 축(모델·전달방식·접근·비용·UX·언어/트리거)으로 따져 골랐다 / 근거는 ADR [0008](decisions/0008-code-review-tool.md)(작성 예정) + 기술 블로그 글.

### Q. 개발 DB와 운영 DB를 어떻게 분리했어요? 왜 그 방식?
- **말하기 좋은 버전**:
  > "처음엔 로컬 개발과 배포된 운영이 같은 Supabase(클라우드 DB) 하나를 봤어요. 그래서 로컬에서 테스트하며 저장한 게 운영을 더럽혔죠. 분리 방식을 셋으로 비교했어요 — ① dev용 클라우드 프로젝트를 하나 더 만들기, ② 클라우드 둘 + 스키마를 SQL 파일로 관리, ③ Supabase CLI로 **로컬에 Docker 스택**을 띄워 dev로 쓰기. 저는 ③을 골랐는데, 로컬에서 인터넷 없이 독립적으로 개발할 수 있고 DB 스키마가 마이그레이션 코드로 git에 남아 포트폴리오로도 좋아서예요. 대신 Docker·CLI 학습 비용이 드는데, 그게 이 프로젝트 목적(도구 체험)과 맞아 감수했습니다."
- 더 물으면: 로컬 Docker 스택이 dev 역할이라 별도 dev 클라우드 프로젝트는 안 만듦(불필요한 건 안 만든다=YAGNI) / 운영·Vercel은 안 건드림 / 근거 ADR [0012](decisions/0012-dev-prod-db-separation.md).

### Q. DB 스키마를 어떻게 관리해요? (마이그레이션 = 변경 기록부)
- **말하기 좋은 버전**:
  > "스키마(표·권한·함수)를 처음엔 Supabase 대시보드에서 클릭으로 만들었어요. 그런데 그러면 '지금 스키마가 어떤 상태인지'가 화면에만 있고 코드(git)엔 없어서 재현이 안 되죠. 그래서 현재 스키마를 통째로 담은 첫 마이그레이션(baseline) SQL 파일로 옮겨 git에 남겼어요. 그러면 로컬 DB도 그 파일 하나로 똑같이 만들어지고, 앞으로 변경도 파일로 차곡차곡 쌓입니다. 그 과정에서 '대시보드가 몰래 해주던 권한 부여(grant)'가 마이그레이션엔 없어 한 번 데었는데, 그 덕에 '코드로 옮길 땐 숨은 자동 처리까지 적어야 한다'를 배웠어요(아래 ③)."
- 더 물으면: baseline은 로컬 provision(빈 DB를 표·함수로 채움) + git 기록 용도, 운영엔 push 안 함(이미 있음) / 다음 변경부터 `db push` + `migration repair`로 운영 동기화.

### Q. velog 글을 어떻게 가져왔어요? (가장 까다로웠던 부분)
- **말하기 좋은 버전**:
  > "velog는 '이렇게 가져가세요' 하는 공개 API 문서가 없었어요. 그래서 직접 찔러보며 방법을 찾았는데, 내부 GraphQL에 글을 요청하니 원본 마크다운이 깨끗하게 오더라고요(HTML을 긁어 파싱하는 것보다 훨씬 안정적). 다만 브라우저에서 직접 부르면 CORS로 막혀서, Next.js 서버 라우트가 대신 호출하게 우회했어요. 비공식 창구라 깨질 수 있으니 한 군데(서버 라우트)에 가두고, 안 되면 마크다운을 직접 붙여넣는 대체 경로도 뒀습니다."
- 더 물으면: RSS·GraphQL·페이지 HTML 세 경로를 실측해 비교했다 / 근거는 ADR [0004](decisions/0004-velog-fetch.md).

### Q. 작업을 어떻게 나눠 PR로 올렸어요? (재설계 경험)
- **말하기 좋은 버전**:
  > "처음엔 한 브랜치에 다 쌓았는데, 그러면 PR 하나가 거대해져서 AI 코드리뷰가 읽을 양이 폭발하고 리뷰도 산만해지더라고요. 그래서 다시 설계했어요 — 이미 테스트로 검증된 토대는 큰 PR 하나로 합치고(리뷰 생략), 그 뒤 UI 작업부터는 기능마다 작은 PR로 쪼개 각각 리뷰받게요. PR을 작게 유지하니 리뷰가 싸고 정확해집니다."
- 더 물으면: 이슈→브랜치→커밋→PR→리뷰→머지 흐름을 `docs/conventions.md`·ADR [0007](decisions/0007-workflow-conventions.md)로 정립.

### Q. 추출기를 실제로 써보니 후보가 노이즈 투성이였어요 (측정 → 원인 분리 → 최소 수정)
- **말하기 좋은 버전**:
  > "완성하고 실제 학습 문서를 올려보니 후보가 151개나 쏟아졌어요. 눈으로만 보면 '추출기가 엉망이다' 싶었지만, 바로 손대지 않고 먼저 측정했습니다. 출처별로 분류했더니 노이즈가 두 갈래더라고요 — ① 세 겹 백틱 코드블록이 통째로 한 후보로 잡히는 '명백한 버그', ② 'Step 5: …' 같은 문장형 제목처럼 '개념이 아닌 정상 추출'. 둘은 성격이 달라서, 누가 봐도 버그인 ①만 TDD로 고치고 ②(무엇을 개념으로 볼지)는 제품 판단이 필요해 일부러 분리했어요. 결과는 151→111개, 멀티라인 노이즈는 거의 사라졌습니다."
- 더 물으면: 원인 ①은 인라인 코드 정규식이 ` ``` `를 탐욕적으로 삼킨 것 / 고친 법은 추출 전에 펜스 블록을 먼저 제거(`/```[\s\S]*?```/g`) / 보너스로 코드블록 *안*의 예시 제목까지 사라져 heading도 40→24로 줄었다.

### Q. 화면 버그를 어떻게 디버깅했어요? ("고장난 빌드를 헛검수한 경험")
- **말하기 좋은 버전**:
  > "고립된 점을 회색·크게 강조하는 기능이 작은 그래프에선 다 똑같이 보였어요. 추측으로 고치지 않고 스크린샷으로 재현부터 했더니 임계값이 원인이었어요 — 기준이 'degree ≤ 1'이라 한 쌍만 이어도 양끝이 '약연결'로 잡혀 회색이더라고요. 그래서 '진짜 고립(degree 0)'만 강조하도록 좁혔습니다. 그런데 더 큰 교훈은 그다음이었어요. 속성명을 weak에서 isolated로 바꾸다 한쪽을 빠뜨려 강조가 아예 꺼진 빌드를, 제가 수동 점검에서 '잘 된다'고 통과시켰거든요. AI 코드리뷰가 그 키 불일치를 잡아줬고요. 캔버스는 자동 테스트가 안 돼 수동 검수가 마지막 보루인데, 그때 '실제로 회색이 나오나'까지 안 보면 놓친다는 걸 체감했습니다."
- 더 물으면: 원인 좁히기 = 스크린샷으로 '연결 안 됨'인지 '임계값 탓'인지 가름 / 고침 = `degree ≤ 1` → 이미 TDD된 `isIsolated`(degree 0) 재사용 / 헛검수 원인 = `node.isolated`가 `undefined`라 늘 falsy → 강조 전부 꺼짐, `tsc`는 콜백이 `any`라 못 잡음 / 교훈 = canvas는 수동 검수 때 "결과물"까지 확인. (이슈 #29)

### Q. 가장 까다로웠던 버그 — 로그인은 되는데 조회만 500
- **말하기 좋은 버전**:
  > "로컬로 분리한 뒤, 로그인은 되는데 그래프 조회(`GET /api/graph`)만 500이 났어요. 추측하지 않고 DB에 직접 들어가(`psql`) 권한을 봤더니, 조회 역할(`authenticated`)에 SELECT 권한이 없더라고요. 운영 대시보드는 표를 만들 때 권한을 자동으로 주는데, 제가 옮긴 마이그레이션 SQL엔 그 grant가 없었던 거죠. 결정적 단서는 '저장(PUT)은 200인데 조회(GET)만 500'이었는데, 저장은 권한을 우회하는 특수 함수(`security definer`)로 돌아서 됐던 거예요. grant를 마이그레이션에 추가해 해결했습니다."
- 더 물으면: 증거 = `information_schema.role_table_grants`에 SELECT 없음 + `set role authenticated; select` → `permission denied` / 교훈 = 스키마를 코드로 옮길 때 대시보드의 숨은 자동 처리도 명시 / 보너스 함정 = `supabase db reset`이 로컬 사용자(`auth.users`)도 지워 401이 떠 재로그인 필요(로컬 한정).

### Q. 분리가 드러낸 잠복 버그 — 운영 로그인이 localhost로 튕김
- **말하기 좋은 버전**:
  > "분리를 끝내고 '운영은 멀쩡한가' 회귀 테스트를 하다가, 배포 사이트에서 로그인하면 엉뚱하게 localhost로 튕기는 걸 발견했어요. 원인은 운영 Supabase의 Site URL이 새 프로젝트 기본값인 `localhost:3000`으로 남아 있던 거였어요. OAuth에서 Site URL·Redirect URLs는 '로그인 후 돌려보내도 되는 주소 허용 목록'인데(아무 데나 보내면 인증 코드를 탈취당하니 막는 보안 장치), 앱이 넘긴 배포 주소가 목록에 없으면 기본값(localhost)으로 떨어지거든요. 운영 Site URL을 배포 도메인으로 고치고 콜백을 목록에 등록해 해결했어요. 분리 작업이 그동안 안 보이던 운영 설정 버그를 드러낸 셈이라 인상적이었습니다."
- 더 물으면: `redirectTo`가 허용 목록에 있으면 그리로, 없으면 Site URL로 폴백 / 로컬은 이제 로컬 스택을 쓰니 운영 목록에서 localhost를 빼 '운영 전용'으로 정리.

### Q. 추출 노이즈를 어떻게 처리했어요? ("고치기 전에 측정")
- **말하기 좋은 버전**:
  > "노이즈를 눈대중으로 고치지 않았어요. 실제 문서를 추출기에 넣어 출처별로 분류(측정)부터 했고, 진짜 원인인 펜스 코드블록만 도려냈습니다. 151→111개, 40자 넘는 긴 노이즈는 다수에서 3개로 줄었어요. '최적화·수정 전에 먼저 측정한다'는 원칙을 실제로 적용한 사례입니다."
- 더 물으면: 수정은 TDD로 — RED(펜스 제외 실패 테스트)→GREEN으로 짜서, 펜스 *밖* 멀쩡한 인라인 코드(`인라인코드`)는 살리고 펜스만 제거됨을 테스트가 보장(회귀 방지) / 근거는 이슈 #10·`extractor.test.ts`.

### Q. 그래프 화면은 왜 테스트(TDD)를 안 했어요? (테스트 계층을 무엇으로 나눴나)
- **말하기 좋은 버전**:
  > "무턱대고 다 테스트로 덮지 않고, '무엇을 무엇으로 검증할지'를 계층으로 나눴어요. 순수 로직(추출기·연결 수 계산)은 유닛 테스트로 TDD하고, 폼류 컴포넌트는 RTL로, 그래프 화면은 사람 눈으로 검수했습니다. 그래프를 테스트에서 뺀 건 게을러서가 아니라 **확인할 방법이 없어서**예요 — 그래프는 캔버스에 픽셀로 그려지는데, 컴포넌트 테스트가 쓰는 jsdom(가짜 브라우저)은 캔버스를 실제로 안 그리거든요. '고립된 점이 회색이고 더 크다'를 테스트 코드로 확인할 길이 없어서, 그건 사람 눈(또는 진짜 브라우저를 쓰는 E2E)의 몫으로 뒀습니다."
- 더 물으면: 종류(유닛/컴포넌트/E2E)·도구(Vitest/RTL/Playwright)·방법(TDD)은 서로 다른 축이라 곱해서 적용 / 검증 가능한 계산(`degree`)은 순수 함수로 떼어 유닛 테스트함(캔버스에서 빼냄) / "선이 보이나"는 RTL 불가·**Playwright(진짜 브라우저)** 영역 / 근거는 spec §9(폼=RTL, GraphView=수동).
- **게이트 강도 연결**: 빌드·타입(약) < 테스트(강) < 사람 눈/E2E(시각·UX). "빌드 통과 = 동작함"이 아니라, 계층마다 잡는 게 다르다.

### Q. 검색·필터는 어떻게 테스트했어요? ("화면에 박힌 로직을 순수 함수로")
- **말하기 좋은 버전**:
  > "검색·주제 필터를 만들 때, 처음엔 거르는 계산을 화면 컴포넌트(page) 안에 그냥 넣었어요. 그런데 그건 React 상태와 엉켜 있어 테스트하기 어렵죠. 그래서 '검색어·주제로 점과 선을 거른다'는 순수 계산만 `filterGraph`라는 함수로 빼서, 테스트를 먼저 쓰고(RED) 구현했습니다(GREEN→REFACTOR). 화면 자체엔 TDD할 게 없지만, 그 안에 숨은 순수 로직은 빼면 테스트할 수 있다는 경계 감각을 배운 사례예요."
- 더 물으면: 추출한 함수 = `filterGraph`(거른 nodes·edges)·`uniqueTopics`(주제 버튼 재료) / "Controls의 파일 다운로드도 TDD?"엔 → `Blob`·`a.click` 같은 브라우저 글루라 불가, 유일한 로직 `exportJSON`은 이미 검증됨 → 수동 검수 / 기준 = "UI엔 TDD 안 함, 숨은 순수 로직은 빼서 TDD".

### Q. 소셜 로그인(OAuth) 흐름을 설명해보세요
- **말하기 좋은 버전**:
  > "구글 로그인은 우리가 비밀번호를 직접 받지 않아요. 흐름은 이래요 — 버튼 클릭 → `signInWithOAuth`를 부르며 '로그인 끝나면 여기로 돌려보내'라는 콜백 주소를 같이 넘겨요 → 구글 로그인 화면 → 사용자가 승인하면 구글이 '이 사람 맞다'는 일회용 코드를 들고 우리 콜백으로 돌아와요 → 콜백 라우트에서 그 코드를 `exchangeCodeForSession`으로 세션(로그인 증표)으로 바꿔 쿠키에 심어요 → 이후 서버 라우트는 그 쿠키를 읽어 '누구냐'를 확인하고 그 사람 데이터만 다룹니다. 비밀번호·계정 보안은 구글에 맡기고, 우리는 코드→세션 교환만 하는 구조예요."
- 더 물으면: 세션을 쿠키로 둔 이유 = 서버 라우트가 읽어야 해서(localStorage는 서버가 못 읽음) / '어디로 돌려보낼지'는 Supabase Site URL·Redirect URLs 허용 목록이 통제(③ 버그 참고) / 앱 코드는 `redirectTo`가 `location.origin` 기반이라 로컬·운영에 자동 대응(코드 수정 없이 로컬 로그인까지 됨).

## 프로세스 스토리 (차별화 포인트)
- 체계적 설계 프로세스: brainstorming → spec → plan → 구현 (superpowers 워크플로우).
- (TDD 도입 시) 테스트 우선 방식을 처음 적용해본 경험과 그 판단 근거.
- 아이디어를 후보 여러 개로 펼치고 트레이드오프로 좁힌 의사결정 과정.

### ★ 선행 규율 — "처음부터 게이트로 굴렸다"
> 코드를 *쓴 뒤* 품질을 붙인 게 아니라, **코드 첫 줄 전에** 규율을 강제로 깔고 시작했다는 스토리.

- **말하기 좋은 버전**:
  > "저는 코드 첫 줄을 쓰기 전에 타입체크·lint·테스트 게이트를 hook으로 강제하고 TDD로 시작했어요. 그래서 (커밋 N개) 내내 게이트가 깨진 적이 없습니다."
- **왜 차별화되나**: 대부분 "나중에 테스트 붙였어요"인데, 이건 "처음부터 규율로 굴렸어요". 품질이 *사후 보정*이 아니라 *기본값*.
- **근거(만들면서 채울 것)**:
  - 깐 게이트: tsc / lint / format / 테스트 러너 — (스택 정해지면 기입)
  - 강제 방식: hook(PreToolUse) 또는 pre-commit — CLAUDE.md "권유"가 아니라 hook "강제"로 한 이유 → `docs/decisions/`(ADR)에 기록
  - 증거: 게이트 통과 커밋 수 / 게이트가 실제로 막아준 사례 → `docs/journal/`에 기록
- **연결**: "약한 게이트(빌드/타입) vs 강한 게이트(테스트)"를 구분해 적용했다 (게이트 강도 인식).

### ★ "결정은 기록해야 사라지지 않는다"
> 다른 프로젝트에서 "왜 이 도구를 골랐지?"를 나중에 못 찾은 경험(기록이 없어서)이 계기.

- **말하기 좋은 버전**:
  > "예전엔 도구를 고른 이유를 안 적어둬서, 나중에 '내가 왜 이걸 썼지?'를 못 찾았어요. 그래서 이 프로젝트는 기술·워크플로 결정을 전부 ADR(결정 기록)로 남깁니다(0001~). 면접에서 '왜 그 선택?'을 물으면 문서로 답할 수 있어요."
- **연결**: 워크플로 컨벤션(브랜치·커밋·PR·이슈)을 `docs/conventions.md` 단일 출처로 + 스킬(`create-issue`·`commit-push`)로 자동화 → "팀처럼 일하는" 프로세스를 솔로에서도 갖춤.
