# 나중에 할 것 (MVP 이후)

> MVP에서 일부러 뺀 것들을 모아둔 곳. **"안 만들기로 한 결정"도 기록**해 둔다 — 나중에 "이거 왜 안 했지?" 하지 않으려고.
> MVP = 관계 발견 그래프(개념 노드 + 손으로 그은 연결선 + 고립 강조) + 검색/거르기.
> **목록 순서 = 우선순위.** (2026-06-04 최초 정함 → **2026-06-05 재배치**: DB 저장·인증을 7→2번으로 앞당김. 근거 [ADR 0010](decisions/0010-db-auth-priority.md).) 먼저 배포해 라이브 산출물을 만들고 → DB·인증으로 풀스택 기반을 깐 뒤 → 추출 품질을 정교화하고 → 능동 학습(퀴즈)으로 키운다.

---

## 1. 배포·공유
- MVP 선택지 4번이었음. 호스팅해서 면접관·블로그 독자에게 보여주기.
- **MVP가 동작한 직후 1순위 마일스톤** (예시 데이터로 배포).
- **가장 먼저 하는 이유**: 싸게 끝나고 포폴 임팩트가 크며, 실제로 써보면 "어떤 글에서 어떤 노이즈가 진짜 거슬리는지" 데이터가 쌓여 **다음 작업의 타깃을 정해준다**(측정하고 최적화).

## 2. 데이터베이스(DB) 저장 + 인증
- MVP는 브라우저 안 저장 + JSON 내보내기 ([ADR 0003](decisions/0003-framework.md)). 서버 DB로 옮기고 로그인(인증)을 붙여, 여러 기기에서 같은 그래프를 쓰고 백업·공유한다.
- **7→2번으로 앞당긴 이유**(2026-06-05, [ADR 0010](decisions/0010-db-auth-priority.md)): 주동기가 **포폴 임팩트**(백엔드·인증·DB 역량을 보여주는 가장 선명한 산출물) + 멀티기기 + 데이터 안전. localStorage가 캐시 삭제로 유실되는 공백을 메움.
- **배포(1번)는 그대로 먼저** — 호스팅·빌드 설정은 DB가 와도 재사용되고, 바뀌는 건 저장 계층(`storage.ts`)뿐이라 선행 비용이 거의 안 버려진다.
- DB는 아래 **퀴즈(능동 학습)의 토대**가 됨 — 이해도를 DB에 저장하면 퀴즈의 "무거움(웹 직접)" 방식이 자연스러워진다. 그래서 퀴즈보다 앞.
- ⚠️ **큰 마일스톤**: 스키마·API·인증·보안·호스팅·비용 → 별도 설계 필요(착수 시 새 ADR 후보).

## 3. velog 전용 똑똑한 키워드 추출 정교화
- MVP에선 일반 마크다운 기준으로 후보를 뽑음 ([ADR 0004](decisions/0004-velog-fetch.md)). velog 줄글에서 핵심 개념만 더 잘 뽑도록 정교화(줄글 특성 반영).
- **우선순위 근거**: 추출 품질은 파이프라인의 맨 앞이라, 후보가 지저분하면 그래프·퀴즈·추천이 모두 그 노이즈를 물려받는다. 펜스 노이즈를 직접 겪고(PR #11) 우선순위를 올림. **퀴즈(큰 피벗) 직전에 입력을 정비**하는 자리.
- 일반 개념 판별 필터(문장형 제목·경로 거르기)는 [ADR 0009](decisions/0009-extractor-noise-strategy.md)에서 보류한 "2단계" — 여기서 함께 재검토.

## 4. 퀴즈로 이해도 측정 (능동 학습 루프) ★ 핵심 아이디어

**무엇**: 그래프의 개념 점(노드)을 클릭하면 → 클로드가 그 개념에 대해 질문 → 내가 답을 타이핑 → 클로드가 맞나·틀리나 판단 → 애매하면 꼬리질문. 그 결과로 그 개념의 '이해도'가 매겨지고, 점의 크기·테두리로 표시된다.

**왜 좋나**:
- "이해도를 누가 판단하나?" 문제를 푼다 — 내가 일일이 자기 평가(번거로움)하는 대신, 퀴즈로 *측정*한다.
- 미뤄둔 "이해도 표시(점 크기·테두리)"가 이걸로 살아난다.
- 정적인 '지도'가 능동적인 '학습 도구'로 진화한다.

**붙이는 방식 두 가지**:
- **가벼움(추천)**: 퀴즈는 클로드코드(CLI)에서 하고, 클로드가 결과(이해도)를 그래프 데이터 파일에 기록 → 웹 앱은 그걸 시각화만. 앱 안에 외부 AI를 안 넣어도 됨.
- **무거움**: 웹 앱 안에 외부 AI(클로드 API)를 넣어 앱이 직접 묻고 판단. 포폴 임팩트는 크지만 큰 작업(API 연동·키·비용). 위 2번(DB)이 깔려 있으면 이해도 저장이 자연스러워진다.

**왜 MVP 아님**: 핵심 가치(관계 발견)와 *다른 가치*(능동 학습)다. 같이 넣으면 초점이 흐려지고 과설계가 된다. MVP(관계 발견 그래프)를 완주한 뒤 얹는다.

**면접 스토리**: "처음엔 되돌아보는 '지도'로 만들고, 다음 단계로 퀴즈를 붙여 이해도를 측정하는 '능동 학습 도구'로 키웠다" = 성장하는 설계.

---

## 5. 이해도 표시 (점 크기·테두리·투명도)
- MVP 선택지 2번이었음. 색은 주제가 쓰니, 이해도는 다른 수단(크기·테두리·투명도)으로.
- 위 4번(퀴즈)과 결합하면 "누가 이해도를 매기나" 입력 부담이 풀린다.

## 6. 다음 학습 추천
- 핵심 가치 후보 중 하나였던 것. 고립·약연결된 점에서 "다음은 이걸 배워봐" 추천.
- MVP의 "고립·약연결 강조"가 이것의 맛보기.

## 7. 연결선(엣지) 자동 추천
- 지금은 연결선을 **손으로** 긋기로 함 ([ADR 0002](decisions/0002-node-source.md) 맥락). 나중에 개념 사이 관계를 기계가 자동으로 제안.
- 가장 어려운 부분이라 일부러 뒤로 미룸.
